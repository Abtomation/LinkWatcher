## Improvement Suggestions

### What worked well
- Step 3 evaluation criteria caught the framework-fit nuance that Update History is a documented state-file pattern in 6 other files; this shaped the deviation note at Step 6 correctly (delete only here, leave the pattern intact elsewhere).
- Self-consistent end-state validation: running the new no-history-append `Update-ProcessImprovement.ps1` to mark its own IMP Completed produced the expected end state with no history row — strongest possible verification for this kind of behavior change.
- Step 11 silent-housekeeping sweep surfaced one substantive Auto-Update Function description in process-framework-task-registry.md that would otherwise have rotted.
- Parse-checking + -WhatIf smoke tests on both modified scripts before committing to writes gave high confidence in the medium-risk classification.

### What could be improved
- Both scripts had near-duplicate Update History append blocks (Add-SingleImprovement vs Add-SinglePilot in New-ProcessImprovement.ps1; pilot vs regular paths in Update-ProcessImprovement.ps1). Each pair required parallel edits.
- The dead `-UpdatedBy` / `-UpdateHistoryNote` / `-ItemUpdatedBy` parameter chain only became apparent after removing the appends. Catching dead-parameter sweeps earlier in planning would have shortened the iteration loop.

### Specific suggestions
- Suggestion (structural, not in scope of this IMP): extract a shared "Append-StateFileHistoryEntry" helper into Common-ScriptHelpers so all state-file scripts that maintain Update History (feature-tracking, feature-request-tracking, architecture-tracking, etc.) call one site. Future policy changes (delete-here-only-not-there scenarios) would then be one-script-edits instead of N. Cost: refactoring 4-5 existing scripts. Benefit: realized only on the next history-related IMP. Note as observation; not advocating an IMP filing — recurrence is unproven.
