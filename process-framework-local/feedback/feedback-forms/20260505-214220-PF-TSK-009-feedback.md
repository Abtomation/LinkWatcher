## Improvement Suggestions

### What worked well

- The IMP cluster (734, 735, 745) all sourced from PF-FEE-1110/1111 was well-scoped because the source feedback forms had concrete drafted examples and named precedents. Step 5 (Read current state) was fast because the same task definition (PF-TSK-087) was the target for 2 of the 3 IMPs.
- The minimum-viability gate at Step 6 surfaced a real alternative for IMP-734 (inline table inside PF-TSK-087 vs. separate guide) and IMP-745 (do-nothing vs. design prompt). For both, the human selected the proposed approach after seeing the alternative — but the surfacing was the load-bearing part, not the outcome.
- Step 13's skip option for low-risk-no-deviation cases let the 3 IMPs share a clean approve-execute-close cadence without redundant final checkpoints.
- New-Guide.ps1's auto-update of PF-documentation-map.md is a model for how scaffolding scripts should integrate with surrounding infrastructure — eliminates a forgettable manual step.
- Update-ProcessImprovement.ps1's atomic move-row-on-Completion is a small but high-value capability — the alternative (manual table edits) is error-prone and the script handles the Update History append in the same transaction.

### What could be improved

- **Single-IMP-style Step 6 ceremony for clustered IMPs**: When multiple IMPs are sourced from the same feedback form and target the same artifact, the per-IMP Step 5 (Read current state) and Step 6 (present analysis + approach) re-introduce context the human just saw. For this session, by IMP-735 the human already knew PF-TSK-087's structure intimately. The task definition could optionally allow batched Step 6 presentation when consecutive IMPs share a target artifact ("present batched analysis for IMPs A + B + C, all targeting X") with one combined approve-or-revise gate. Trade-off: lose per-IMP rejection clarity. May not be worth the structural change.
- **Step 12's `--new-tool` surprise**: feedback_db.py rejected `PF-TSK-087` as unknown despite the task being live for one prior session. The guard is correct behavior, but it suggests the feedback DB tool index is not auto-populated when New-Task.ps1 creates a task. A small one-liner integration in New-Task.ps1 would close this gap.

### Specific suggestions

1. (Possible — low priority): Add batched Step 6 mode to PF-TSK-009 for clustered IMPs targeting the same artifact, with explicit caveats about when to use it (only when human is already in context for the target). Could be a single sub-bullet under Step 6 rather than a structural change.
2. (Possible — low priority): Auto-register newly created tasks in feedback_db.py's tool index from New-Task.ps1 to avoid the `--new-tool` surprise on first feedback log.
