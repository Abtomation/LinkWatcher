## Improvement Suggestions

### What worked well
- **Step 6 minimum-viability gate fired correctly for IMP-732**, prompting genuine A/B/C scope deliberation rather than auto-accepting the "armor all 8" framing in the IMP description. The user explicitly chose Option A with a confidence vote ("you will manage in one session") — the framework surfaced the choice cleanly.
- **Risk-classification at Step 10 scaled to scope**: Low-risk single-edit (IMP-726), Medium-risk batch (IMP-732, 8 scripts), Low-risk mechanical removal (IMP-733). Per-change checkpoints would have made IMP-732 unworkable; bulk + grep + parser verification was correct.
- **Step 11 sweep produced substantive findings exactly twice** (IMP-726: temp-process-improvement-state-template.md duplicated convention summary; IMP-733: script-soak-tracking.md Notes column referencing pre-cleanup state). Stayed silent for IMP-732 where references were about script *functionality* (unchanged) rather than internal armoring state. Right granularity.
- **Mid-session validation of IMP-726's clarification**: the canonical TOOL_DOC_ID convention clarified at Step 12 (what counts as a "task definition" — file with `id: PF-TSK-NNN` frontmatter) was applied for IMP-732 and IMP-733 logging within the same session. The fix paid off immediately.

### What could be improved
- **IMP-732 scope-decision evidence**: the IMP description listed 8 scripts in scope based on prior session's PF-PRO-028 v2.0 inclusion-criteria audit, but didn't surface that audit's per-script reasoning. When evaluating Option C (re-audit before armoring), I had no easy way to see the prior audit's classification per script. This isn't a process gap — the prior session's state file (now in `old/`) likely has that detail — but a "see PF-IMP-728 state file Section X" pointer in IMP-732's description would have helped.
- **feedback_db.py `--new-tool` atomic-batch behavior**: when batch-logging 8 changes for IMP-732, 2 first-time tool registrations caused the entire batch to fail. Solution (`--new-tool` retry) was clear, but resulted in no partial logging on first attempt. For a single-shot command this is fine; for an automation pipeline it would be friction. Keeping the safety check is right; the docs already warn about it.

### Specific suggestions
- For multi-script IMPs filed during a "follow-up" pattern (like IMP-732 from IMP-728), include explicit cross-references to the prior session's state file or audit data so the executing agent can verify the per-script in-scope reasoning without re-deriving it.
- *(Observation only, not a fix request)* The IMP-733 description said "twice" but actual counter drain was 3x (1 from script + 2 from helper for the 2 helper functions called). A minor inaccuracy in the IMP description but didn't affect execution.
