## Improvement Suggestions

### What worked well
- Step 2 evaluation table with minimum-viability gate caught real scope creep on IMP-616 (17 → 6 conversions for ~80% UX benefit).
- Step 8 fast-path skip directive (PF-IMP-594) saved one round-trip per IMP without losing checkpoint discipline.
- Bulk-changes guidance in Step 9 ("apply all then grep-validate completeness") matched the IMP-616 mechanical rollout perfectly.
- feedback_db.py log-change --batch with --new-tool flag handled mixed populations cleanly.
- Update-ProcessImprovement.ps1 transitions (NeedsPrioritization → InProgress → Completed) all worked first-try.

### What could be improved
- **Bash backtick gotcha in -ValidationNotes**: When invoking Update-ProcessImprovement.ps1 from bash, backticks around inline-code references (e.g., `` `[string]$BatchFile` ``) are interpreted as bash command substitution, silently truncating the literal-code spans in the validation notes. The script ran "successfully" but stored corrupted notes. I had to use Edit to repair the tracking file.

### Specific suggestions
- Add a one-line warning to PF-TSK-009 Step 15's Update-ProcessImprovement.ps1 example: "When invoking from bash, prefer single-quoted -ValidationNotes (no `$` or backtick interpretation) or escape backticks. Backticks in bash trigger command substitution and silently truncate literal-code references."
- Or: add a parameter-level note to Update-ProcessImprovement.ps1's .PARAMETER ValidationNotes block flagging this same gotcha.
- The single-item-unwrap quirk in New-ProcessImprovement.ps1 -BatchFile (1-item JSON arrays get unwrapped by ConvertFrom-Json) is a candidate follow-up IMP — file `@($items)` re-wrapping defensively. Not in IMP-617 scope but observed during testing.
