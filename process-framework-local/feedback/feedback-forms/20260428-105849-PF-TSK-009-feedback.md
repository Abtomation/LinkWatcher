## Improvement Suggestions

### What worked well
- **Step 2 evaluation table forced a real rejection**: IMP-601 was filed as a placeholder for future work ("would be cleaner") with no concrete pain point. Without the Minimum-Viability + Recurring-Value + Complexity-to-Benefit criteria forcing explicit rating, my default behavior would have been to find some way to start implementing it. The table caught it as an over-engineered placeholder and routed to rejection.
- **Step 11 grep sweep caught scope gaps**: For IMP-602, the IMP description listed 4 files. The mandatory grep sweep surfaced 2 additional active framework files (`temp-state-tracking-customization-guide.md`, `framework-evaluation-map.md`) that needed updating. Without this step, those files would have shipped with stale references.
- **Session limit (3 IMPs) prevented over-extension**: After IMP-605 completed, the rule "proceed directly to finalization (Step 17/18)" auto-routed me to the feedback form rather than fishing for a 4th IMP. Quality of finalization steps stayed high because context budget remained available.
- **Routing rules in Step 2**: The "content update / structural change / new framework capability" trichotomy correctly routed all 3 IMPs to PF-TSK-009 itself. None needed delegation. The routing question is fast to answer and prevents scope creep.
- **Reflexive bug fix**: IMP-605's fix to Validate-FeedbackForms.ps1 enabled the same script to validate PF-FEE-1005 (this form) at session close. End-to-end loop closed in one session.

### What could be improved
- **Step 12 tool ID convention is undocumented**: The `<TOOL_DOC_ID>` placeholder in PF-TSK-009 Step 12 doesn't disclose that tasks use `PF-TSK-NNN` form while non-task documents use filename form (`framework-evaluation-map.md`). My first batch invocation failed because I used registry IDs (`PF-GDE-011`) for non-task docs. The error message correctly pointed to `list-tools` for diagnosis, but the convention itself should be in PF-TSK-009 Step 12.
- **Step 9 "verify completeness with grep-based checks" lacks a concrete pattern**: For bulk/repetitive changes, the instruction is correct but vague. A boxed example (e.g., "After applying replacements, run `Grep -pattern OLD_VALUE -path SCOPE_DIR` and confirm 0 hits") would shave a step.
- **PF-IMP-603 friction is multiplicative**: every IMP whose row content contains literal `|` characters now requires the manual `&#124;` workaround before Update-ProcessImprovement.ps1 will run. Hit twice in 3 IMPs this session (IMP-605 directly, IMP-597 historically). Worth elevating priority once the fix is scoped.

### Specific suggestions
- Add to PF-TSK-009 Step 12 a note: "**Tool ID convention**: For task definitions use the task ID (e.g., `PF-TSK-009`); for templates, guides, scripts, and context maps use the filename including extension (e.g., `framework-evaluation-map.md`, `Validate-FeedbackForms.ps1`). When in doubt, run `python process-framework/scripts/feedback_db.py list-tools | grep <hint>`. Use `--new-tool` flag to register a previously-untouched canonical document."
- Add to PF-TSK-009 Step 9 (sub-bullet under "For bulk/repetitive changes"): "Concrete verification pattern: `Grep -pattern <OLD> -path <SCOPE>` should return zero hits. Present the empty result as evidence at Step 13 checkpoint."
- Update Update-ProcessImprovement.ps1's pipe-parser error message to include a remediation hint: "Apply HTML-entity workaround: replace literal `|` in the row's text with `&#124;`. See PF-IMP-603."

## Additional Context

### Task-specific challenges
- **Heterogeneous IMP types in one session**: This session covered three distinct IMP shapes — a rejection (IMP-601, no implementation), a sweep across multiple files (IMP-602, mechanical replacement with one special case), and a script bug fix (IMP-605, root-cause-and-repair). PF-TSK-009 absorbed all three with the same Steps 1–18 structure; only the contents of Steps 9–12 varied. The single-task-shape was the right design choice — a per-IMP-type task split would have fragmented the workflow without value.
- **Audit-trail vs active-framework boundary**: For IMP-602, deciding which files to sweep required distinguishing "active framework guidance" (sweep) from "historical audit records" (preserve). The IMP description listed only active framework files, which was the correct scope, but the rationale for excluding `process-framework-local/feedback/**` and IMP audit history wasn't explicit anywhere — I had to derive it from first principles (rewriting historical records breaks the "what did we believe at the time" property). Worth capturing as a guideline for future sweep IMPs.
- **PF-IMP-603 workaround friction**: Hitting the pipe-parser bug mid-claim of IMP-605 required pausing IMP-605 work to apply the workaround to its tracking row. The workaround is documented in PF-IMP-603's history but not in the task definition — a future agent without prior session memory would have to discover it.

### Integration with other tools
- **Update-ProcessImprovement.ps1 ↔ process-improvement-tracking.md**: The script is the authoritative writer for status transitions. Direct file edits to status would bypass summary-count updates, history appends, and frontmatter date — the workaround for the pipe-parser bug had to be applied as a row-content edit (legitimate) and then the script re-run for the actual status transition (legitimate use). Clean separation of concerns.
- **feedback_db.py ↔ feedback forms ↔ process-improvement-tracking.md**: Three-way relationship works cleanly. IMP completion writes to tracking file (operational); tool change records write to feedback DB (analytics); feedback form (this document) writes to per-session feedback forms (qualitative review). No surface duplicates another. The convention boundary (task IDs vs filenames in the DB) is the only friction.
- **Validate-FeedbackForms.ps1 ↔ Common-ScriptHelpers/Core.psm1**: The IMP-605 fix introduced a dependency on `Get-ProjectRoot` from Core.psm1, exposed via the Common-ScriptHelpers facade. The walk-up module loader at lines 18–27 of the script already imports the facade, so `Get-ProjectRoot` is in scope without additional import statements. Facade pattern absorbed the new dependency cleanly.
- **LinkWatcher**: All file edits in this session (6 files in IMP-602 + 1 file in IMP-605) were detected and any internal links updated automatically. No manual link maintenance required.
