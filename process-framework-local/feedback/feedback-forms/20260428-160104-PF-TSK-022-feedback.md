## Improvement Suggestions

### What worked well
- Step 1's "Independently verify tech debt descriptions" warning kept me from accepting the TD's incorrect rationale ("Phase 2 prefix-match implicitly handles parent refs"). The fix would have been technically right but documented for the wrong reason.
- L2 DA-category guidance (drift root-cause trace) reframed the work — the root-cause analysis became the plan's most useful section, not the 9-line edit.
- L8 Documentation-only shortcut compressed a multi-grep checklist into one batched N/A line.
- New-RefactoringPlan.ps1 -DocumentationOnly variant: tailored template, no test/perf sections to skip.

### What could be improved
- The TD228 description in [technical-debt-tracking.md](/doc/state-tracking/permanent/technical-debt-tracking.md) included an incorrect rationale ("handled implicitly by the Phase 2 prefix-match branch") that survived from creation through to refactoring. This is not a tool issue — it's a TD-authoring issue. When DA tech debt is recorded with a proposed-fix rationale, that rationale is rarely re-verified before the refactoring session, and an incorrect rationale would propagate into a fix's framing if Step 1's verify-warning isn't acted on.
- The Documentation-only Refactoring Plan template's "Refactoring Strategy → Implementation Plan → Phase 1 / Phase 2" structure is heavier than needed for single-document/single-edit DA fixes. Filled out for this 9-line edit, the Implementation Plan reads as overstructured. Consider an even-lighter template variant, or guidance that single-edit DA fixes can collapse Implementation Plan to one line.

### Specific suggestions
- **Bug-Triage / TD-authoring guidance update (PF-TSK-016 Bug Triage or wherever DA tech debt is created)**: When recording a DA item that proposes a "handled by X" rationale, the recorder should briefly verify the proposed handler actually does the claimed work — a 30-second grep would have caught TD228's "implicit handling via Phase 2" inaccuracy at recording time. Could be a one-sentence addition: *"For DA items, briefly trace the proposed-fix rationale in the actual code; an unverified rationale here will mislead the refactoring session unless they re-verify."*
- **Documentation-only template lite variant (or template guidance)**: For single-document/single-edit DA fixes (the most common DA shape), the Implementation Plan section's "Phase 1 / Phase 2" scaffold is the wrong shape. Consider either: (a) a `-MinimalDocumentationOnly` flag that drops the Phased structure, or (b) explicit guidance in the template that single-edit fixes can write `Implementation Plan: One edit — see Affected Documents above`.
