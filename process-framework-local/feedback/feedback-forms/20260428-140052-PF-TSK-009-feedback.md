## Improvement Suggestions

### What worked well
- Step 2 evaluation table caught IMP-614's flawed premise via the data-driven validation row; without it I likely would have implemented a small change that broke validation of 30+ files
- Step 11 grep sweep with mandatory hit-list table caught PF-documentation-map.md update needs in both IMP-610 and IMP-615 (would have been missed otherwise)
- Allowlist design pattern for Check-GitObjectsLiteral.ps1 — the IMP-590 narrative explicitly named the 3 forensic files, which fed straight into the script's allowlist constant
- Pre-existing test infrastructure (`.pre-commit-config.yaml` already had a `repos: - repo: local` example with pytest-quick) made IMP-615 implementation a trivial mirror operation
- Update-ProcessImprovement.ps1 supports the full lifecycle (InProgress, Completed, Rejected) with appropriate required-parameter discipline (Impact required for terminal states only)

### What could be improved
- For rejected IMPs, Steps 7–14 of PF-TSK-009 are largely no-ops (nothing was implemented to review). Could benefit from a "rejection short-circuit" path that goes Step 5 → Step 15 (record rejection) → Step 17, skipping the implementation/verification ceremony that doesn't apply
- feedback_db.py batch mode can't apply `--new-tool` per-item; if a batch has even one new tool, must pivot to single-call mode for that item

### Specific suggestions
- Consider adding a "Rejection Path" subsection to PF-TSK-009 Step 5 that lists the explicit short-circuit: Step 5 reject → Update-ProcessImprovement.ps1 -NewStatus Rejected → next IMP or finalize. Would save ~3 cognitive checkpoints for clear-reject cases
- (Out-of-scope for this IMP, just an observation) The IMP-595 "Mechanical Rename" change-type for PF-TSK-014 — if that lands, IMP-614 could be re-filed as a Mechanical Rename touching template + guide + 28 reports + regex in one task, finally making the canonicalization actionable
