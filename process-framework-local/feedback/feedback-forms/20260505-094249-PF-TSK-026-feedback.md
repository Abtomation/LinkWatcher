## Improvement Suggestions

### What worked well
- PF-TSK-026 Step 4.5 Pilot vs Full Rollout Decision — gave me a concrete framework to defend the Full Rollout choice rather than implicit decision-making.
- "ONE PHASE PER SESSION" rule with the Phase 1-2 collapse-allowance — I needed to do planning across two phases in one session and the rule's rationale ("checkpoint discipline + Session Tracking labeling honesty") let me reason from principle.
- Mid-session smoke-test gate against a known-historical-bug surface (the 2026-04-27 _Test-CallerWhatIf module-isolation incident) — the discipline of building disposable synthetic callers + running 5 explicit tests + presenting results before continuing is exactly the gate that would have caught the original bug. Worth promoting from one-task best-practice to framework guidance.
- LinkWatcher's automatic relative-path rewriting on file move — the concept doc went from `proposals/old/` to `proposals/` and 7 inbound references plus all internal `../../../` paths inside the moved file were updated transparently.
- Filing IMPs via `New-ProcessImprovement.ps1` mid-task: PF-IMP-728 (this work's IMP) and PF-IMP-729 (companion lifecycle gap) both got tracked properly without disrupting the task flow.

### What could be improved
- New-StructureChangeState.ps1 -ChangeType "Framework Extension" template hardcodes "Session 1: 2026-05-19" / "Session 2: 2026-05-19" dates instead of "TBD" placeholders.
- The same template's "Phase 1: Preparation (Session 1)" / "Phase 2: Create & Modify Artifacts (Session 1-2)" / "Phase 3: Validation & Cleanup (Session 2-3)" headings inside the Implementation Roadmap section collide with PF-TSK-026's own Phase 1-4 numbering, creating naming ambiguity in customization.
- PF-TSK-026 Step 2 doesn't address the "modify previously-resolved concept doc" case explicitly. The script-self-verification.md concept was archived in `proposals/old/` after the v1.x pilot resolved (PF-IMP-688); this v2.0 work needed the doc restored to `proposals/`, not a brand-new concept created. Step 2 implicitly assumes new-concept creation.

### Specific suggestions
- Edit `process-framework/templates/support/structure-change-state-framework-extension-template.md` to replace hardcoded session dates with "TBD" placeholders, mirroring the placeholder style used elsewhere in framework templates.
- Edit the same template to rename internal "Phase 1/2/3" headings to "Stage 1/2/3" or "Step Group 1/2/3" to avoid collision with PF-TSK-026's Phase numbering. (This is doc-fixable; not memory material.)
- Add a short callout to PF-TSK-026 Step 2: "If revising a previously-resolved concept doc archived in `proposals/old/`, restore via `git mv` (or `mv` if not git-tracked) and bump version + add Revision History subsection — do NOT create a new concept doc via `New-FrameworkExtensionConcept.ps1`."
- Consider promoting the mid-session smoke-test gate pattern to a reusable framework practice in PF-TSK-026 Phase 3 guidance: "When the implementation modifies a module that has previously regressed (consult Lessons Learned in any associated concept doc), insert a mid-Phase-3 checkpoint that builds disposable synthetic callers exercising the canonical regression conditions before proceeding to broader rollout."

### Task-specific challenges
- The work was a reversal of a previously-Resolved IMP decision (PF-IMP-688). Handled by filing a new IMP (PF-IMP-728) with explicit "reverses PF-IMP-688" framing in Source, plus appending an audit-trail note to the original PF-IMP-688 row in Active Pilots (deferred to Session 2 since the row also needs to migrate to Completed Improvements per PF-IMP-729). The framework didn't fight this, but it didn't actively help either — there's no formal "reversal" IMP type.
- Session 1's mid-session smoke-test gate consumed ~10 minutes and produced no deliverable code, but was the highest-value step of the session. It both proved the v2.0 caller-aware mode worked AND verified the historical 2026-04-27 _Test-CallerWhatIf module-isolation bug does NOT regress under the new code path. Without this gate, broad rollout would have built on unverified foundation.

### Integration with other tools
- ExecutionVerification.psm1 v2.0 changes integrate with `_Test-CallerWhatIf` (existing fix for the 2026-04-27 module-isolation incident) — the new `_Resolve-CallingScript` private helper uses the same `Get-PSCallStack` mechanism, so any bug in callstack resolution would affect both. The smoke-test gate verified both paths still work.
- The two-IMP filing (PF-IMP-728 + PF-IMP-729) integrates with the upcoming PF-IMP-729 work via `Update-ProcessImprovement.ps1`'s ability to move pilot rows — when PF-IMP-729 implementation lands, PF-IMP-688 should migrate from Active Pilots to Completed Improvements section (carrying the audit-trail note appended in Session 2 of this work).
- The git-mv concept-doc restore from `proposals/old/` to `proposals/` interacted with LinkWatcher cleanly — 7 inbound references updated automatically, all internal relative paths in the moved file rewritten transparently. This is the kind of "boring infrastructure that just works" that's easy to forget but worth acknowledging.
