TD075 was reported by **PD-VAL-053** (Data Integrity Validation, task PF-TSK-076) during Round 2 validation. The item flagged that `last_scan` property in `LinkDatabase` is not protected by `self._lock`, noting "Low risk due to CPython GIL but not formally thread-safe." The reporting was technically accurate — the property does lack explicit locking — but the recommendation to "wrap getter/setter in `with self._lock`" was not viable because `clear()` already holds `self._lock` when calling `self.last_scan = None`, which would cause a deadlock on the non-reentrant `threading.Lock`.

**Decision**: Won't Fix. The GIL guarantees atomicity for single-reference read/write, the worst-case on GIL-free Python is a stale timestamp (no corruption), and the fix would require either switching to `RLock` (changes concurrency semantics) or breaking the property abstraction — both adding complexity for zero practical gain.

## Improvement Suggestions

### What worked well
- TD075 description in technical-debt-tracking.md was clear and included the key context ("Low risk due to CPython GIL") that enabled quick analysis
- Assessment ID (PD-VAL-053) provided clear traceability to the reporting validation session

### What could be improved
- **Reporting task (PF-TSK-076 Data Integrity Validation)** suggested a fix ("wrap getter/setter in `with self._lock`") without checking whether that fix would actually work. The `clear()` method already holds `self._lock` when calling `self.last_scan = None` — adding a lock to the property setter would deadlock. Validation tasks that recommend fixes should verify the proposed fix is viable, not just identify the symptom.
- More generally, threading-related items in validation should consider whether the existing concurrency model (GIL, lock type) makes the issue moot before reporting it as debt.

### Specific suggestions
1. **Data Integrity Validation task (PF-TSK-076)**: Consider adding a checkpoint in the validation process for threading-related findings: "Have you verified the proposed fix doesn't conflict with the existing locking strategy?" This would catch deadlock-prone recommendations before they become tech debt items.
2. **Tech debt reporting general**: When a TD item includes a "Fix:" suggestion, the reporter should verify the fix compiles/runs at minimum. A non-viable fix suggestion wastes the refactoring session's time re-analyzing from scratch.

### Room for improvement in reporting task
The Data Integrity Validation task correctly identified a formal thread-safety gap, but the recommendation was surface-level. For concurrency findings specifically, the validation dimension should require the reporter to:
1. Check which lock type is used (`Lock` vs `RLock`)
2. Trace all call paths that mutate the flagged state under lock
3. Verify the proposed fix doesn't introduce deadlock or require lock type changes

This would filter out low-value items before they enter the tech debt pipeline.
