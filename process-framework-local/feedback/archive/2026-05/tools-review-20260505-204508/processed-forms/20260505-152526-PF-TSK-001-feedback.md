## Improvement Suggestions

### What worked well
- Auto-registration of new task into ai-tasks.md, PF-documentation-map.md, and PF-id-registry.json — zero manual sync required
- Temp state TaskCreation variant naturally bridged to the task creation step (Phase 1 entry pre-populated the command)
- Forced separation of "create the task definition" (script) from "customize the task content" (manual edits) prevented half-baked auto-generated task bodies
- The script's post-creation message about "MANDATORY NEXT STEP: Task Creation Guide Review" is a useful prompt even if I didn't follow up with the guide read this session (the user gave content guidance directly)

### What could be improved
- Heavy 5-phase template overhead when the new task is definition-only (no new templates/scripts/IDs/context map)
- `-?` help should expose `ValidateSet` allowed values, not just parameter names
- Two PowerShell invocation patterns coexist in the framework (`cd && .\Script.ps1` vs `pwsh.exe -ExecutionPolicy Bypass -File`) — the temp state template uses the first, CLAUDE.md prefers the second
- No structured prompt during task design to consider durable-vs-transient state separation — the user had to manually surface this requirement (sync backlog as durable working document, separate from per-session temp state)
- Cross-repo relative paths (e.g., from a LinkWatcher task to a FrameworkBuilder file) are easy to miscount; would benefit from a quick sanity check pattern in the task creation guide

### Specific suggestions
- Add a lightweight TaskCreation variant (e.g., `-Variant TaskCreationDefinitionOnly`) for tasks that don't introduce new file-creation infrastructure — drops Phases 2–4 (Document Creation, Templates, Visualization)
- Update `-?` help in `New-TempTaskState.ps1` and `New-Task.ps1` to print `ValidateSet` values, e.g., via `Write-Help` block with `Variant accepts: TaskCreation, ProcessImprovement, FrameworkExtension`
- Update the temp state template's Phase 1 command line to use the `pwsh.exe -ExecutionPolicy Bypass -File` pattern matching CLAUDE.md
- Add a checklist item to PF-TSK-001's task creation guide: "Does this task have both per-session and cross-session state needs? If yes, design distinct durable + transient artifacts."

## Additional Context

### Task-specific challenges
- The task being created (framework-blueprint-sync) is itself a meta-task spanning two repos (LinkWatcher source + FrameworkBuilder blueprint). The framework's task creation infrastructure is built primarily for tasks that operate within a single project, so cross-repo concerns (relative paths, framework variant resolution, the project ↔ blueprint relationship) had to be invented in the task definition rather than drawn from existing patterns.

- Manual `Edit` and `Write` were the workhorses for content customization; this is expected since automation can only scaffold structure, not the task's actual logic
