## Improvement Suggestions

### What worked well
- **Smoke-test gate pattern**: Mid-session verification with synthetic callers (in both Session 1 for caller-aware mode and Session 2 for helper-routed armoring) caught zero regressions but provided high confidence to proceed. The pattern is genuinely worth its overhead when modifying classes of code that have bitten the project before (here: the 2026-04-27 module-isolation -WhatIf bypass bug).
- **Caller-aware no-arg mode + Pattern B helper-routed armoring**: scaled one helper-module edit to cover ~50 calling scripts. The "register once per script, helper does the work" model is exactly right for this kind of cross-cutting concern.
- **Defensive try/catch around helper-side soak calls**: Ensures soak-side issues never mask the actual helper's behavior. Minor but important detail.
- **Idempotent Register-SoakScript** (silently no-ops on already-registered): Made Pattern B opt-in at top of every script body safe and reusable.
- **PF-PRO-030 Active Pilots audit-trail note approach**: Status NOT flipped, just appending a chronological note linking to the reversing IMP — preserves history, defers the formal lifecycle migration to a separate IMP. Cleaner than retroactively editing the resolved decision.
- **Two follow-up IMPs filed at Phase 4** (PF-IMP-732, PF-IMP-733): Clean scope discipline — defer non-essential cleanup to focused IMPs rather than expanding session scope.

### What could be improved
- **Group C update scripts (inline body, no Main fn) are awkward to armor**: Wrapping the entire body in try/catch (including helper function definitions) is functionally correct but visually ugly. Could be addressed with: (a) a refactor toward consistent Main-fn skeleton across all update scripts, OR (b) a documented "Lite Pattern A" variant (preamble + success Confirm only, no full wrap) for scripts where the wrap is disproportionate.
- **Behavioral change under -WhatIf**: Armored helpers now FAIL LOUDLY because Set-Content honors WhatIf and writes nothing, then Assert-LineInFile catches the missing file. Pre-armoring, the same scenario silently "succeeded" (returned `$true` despite no actual write). Strictly an improvement (silent-success → loud failure) but it's a behavioral change. Worth either: (a) updating the helper to short-circuit early under -WhatIf, OR (b) documenting the new behavior.
- **2 of 13 update scripts didn't import Common-ScriptHelpers**: Update-LanguageConfig.ps1 and Update-WorkflowTracking.ps1 needed the standard walk-up import block added. This was unexpected — implies inconsistent script-template adoption across the framework. A linter check could surface scripts missing the standard import.
- **Validate-StateTracking.ps1 baseline has 4 pre-existing errors**: These obscure regression signal — I had to manually verify none of the errors mentioned my edited files. A cleaner baseline (or a `--show-only-new-errors` flag) would make regression detection easier.

### Specific suggestions
1. **`Apply-SoakArmor.ps1` helper script**: Generalize the one-off batch armorers from Steps 3 and 4 into a reusable script that takes a script path + Pattern (A/B) and applies caller-aware no-arg armoring idempotently. Reusable for PF-IMP-732 (8 outliers) and future armoring work.
2. **Documented "Lite Pattern A" variant**: For scripts where full try/catch wrap is disproportionate (Group C without Main fn), document a lighter alternative that just adds preamble + success Confirm. Trades failure-reset semantic for less invasive code.
3. **Consider helper-side -WhatIf short-circuit**: New-ProjectDocumentWithMetadata / etc. could detect WhatIf at entry and skip both Set-Content and Assert, returning $true without doing real work. Restores pre-armoring -WhatIf behavior.
4. **Linter check for missing Common-ScriptHelpers import**: Surface scripts that should import the helpers but don't (e.g., Update-LanguageConfig.ps1, Update-WorkflowTracking.ps1).
5. **Validate-StateTracking.ps1 `-NewErrorsOnly` flag**: Compare current run against a baseline to surface only newly-introduced errors. Would make regression checking trivial.

## Additional Context

### Task-specific challenges
- **Heterogeneous script structures across 13 update scripts**: Probing revealed 3 distinct patterns (Main fn + simple invocation, existing top-level try/catch, inline body with no Main). Required group-specific armoring strategies. The probe script approach (write a one-off PS analyzer that classifies each script) was the right way to handle this.
- **Cross-module call resolution under armoring**: When DocumentManagement.psm1 calls Test-ScriptInSoak (defined in ExecutionVerification.psm1), how does PowerShell resolve the function? The pattern works because the umbrella facade (Common-ScriptHelpers.psm1) loads both sub-modules and the calling script imports the facade — but per the Script Development Quick Reference, the canonical pattern is to add explicit imports at the sub-module top. I added them defensively.
- **Defensive try/catch design choice**: For helper-side soak calls (Pattern B), wrap defensively so soak errors don't mask the helper's `$true/$false` contract. For script-side soak calls (Pattern A), don't wrap defensively — if soak fails at the script level, the script SHOULD fail loudly. Two different conventions for two different layers; took some thought to get right.

### Integration with other tools
- **`New-ProcessImprovement.ps1`**: Filed 2 follow-up IMPs (732, 733) cleanly. The script's `-Source`/`-SourceLink`/`-Notes` parameters captured the deferred scope precisely.
- **`Update-ProcessImprovement.ps1`**: Marked PF-IMP-728 Completed with detailed `-ValidationNotes` summarizing the cross-session work. The Completed Improvements section now has a reasonable historical record.
- **`Validate-StateTracking.ps1`**: Used as regression check at end of Step 8. Baseline-vs-current comparison was manual; a `-NewErrorsOnly` flag would make this trivial.
- **`script-soak-tracking.md`**: The shareable state file pattern (PF-SST prefix, `process-framework/state-tracking/permanent/` location) worked correctly across session boundaries. The Update History annotation convention (preserve smoke-gate entries with annotation rather than delete them) made it possible to reconstruct what was tested.
