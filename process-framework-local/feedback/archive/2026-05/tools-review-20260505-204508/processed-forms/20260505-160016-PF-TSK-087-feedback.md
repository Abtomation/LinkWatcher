## Improvement Suggestions

### What worked well

- The "Per-Directory Handling Rules" table at the top of the task definition is the most useful single artifact in the task — it lets the agent classify ~90% of items deterministically and routes the remainder to the user.
- AI Agent Role's "default to asking" instruction directly caught the ratings.db misclassification at the checkpoint instead of after deletion.
- The split between durable (backlog + log per variant) and transient (temp state per session) matches the actual lifecycle: most state belongs across sessions; only the in-flight selection is one-session-only.
- The backlog + log structure scaled well across multiple addendums: each catch-up round appended a "Synced 2026-05-05 (addendum N — ...)" subsection without disrupting the durable narrative. Provides a natural audit trail of "what we discovered when, and why."
- Step 5's discovery sub-bullet "Files/dirs present in blueprint but missing in project — flag — likely stale blueprint or project-removed; ask user" was exactly the right framing for the `pre-framework/` and `technical-debt/{assessments,debt-items}/` cases. Same framing also worked for DOC-7 (`PD-ARC`/`PD-AIA`/`PD-TEC`/`PD-FEA` retained as framework-canonical scaffolding after verifying script references).
- Verifying prefix references via grep before deciding RETAIN vs REMOVE on DOC-7 was the right move — would have been easy to wrongly classify those 4 prefixes as stale and propose deletion. The "is it referenced?" check is a simple, durable validation.

### What could be improved

- **The discovery step (Step 5) needs a depth heuristic.** This is the single biggest issue surfaced by the session. My initial walk for `process-framework-local/` was 1 level deep ("subdirs match → no drift"). That missed PFL-2/3/4/5. After the user flagged it, I deepened the walk for that one directory but then declared `test/` and `doc/` mostly-clean based on a similar shallow view. That missed TEST-6/7/8 + DOC-5, and during the deep walk for those, I surfaced DOC-7→11. **The pattern is: skeleton files (registries, trackers, doc-maps) carry framework-shape that needs section/table/schema-level comparison, not just file-presence comparison.** The per-subdirectory consideration policy doc the user requested (see Process Improvements) directly addresses this.
- `New-TempTaskState.ps1 -Variant TaskCreation` produces a template that does not fit a sync session at all (Phases 1–5, Placeholder Components, etc.). Either add a `BlueprintSync` variant or change PF-TSK-087 Step 7 to use a different mechanism.
- The validation rule "No project-specific identifier leaked outside `process-framework/`" treats blueprint root files as a binary "leak or not" decision. The ratings.db case shows there's a third category: intentional cross-project artifacts (shared databases, central registries) that legitimately live at blueprint root. The validation rule should distinguish these from leaks, and the per-item classification step should explicitly flag any populated DB files for user confirmation before any DELETE action.
- Step 14 "Update source project tracking" is mentioned but doesn't say where. PF-STA-103 was the natural target here, but a future agent without that context might miss it. Consider: "if the source project's temp states or IMPs flagged the synced changes, find them via grep for the blueprint path and mark them synced."
- I did not call `get_current_time` MCP at session start, so session duration is estimated. The `.ai-entry-point.md` Step 2 explicitly requires this — adherence gap on my part. Compounded by the session running ~3x longer than originally estimated.
- **The discovery agent I delegated to twice gave at-best-OK reports both times.** First time it stopped at the 1-level subdir comparison; second time it had a false positive on `PD-documentation-map.md` section drift (the agent reported "Blueprint does NOT have `## ci-cd/` section that exists in project docs" which turned out to be wrong — both maps have identical section structure). For sync-style structural comparison, agent reports need to be sanity-checked, especially when they claim drift in places the user wouldn't expect. I would have caught the false positive earlier if I'd grepped section headers myself before reporting.

### Specific suggestions

1. **(USER REQUEST 2026-05-05)** Create a per-subdirectory consideration policy doc and link it from PF-TSK-087 Step 5. Concrete example entries: `state-tracking/temporary/` → ignore; `state-tracking/permanent/` → always consider (deep skeleton-file walk required); `feedback/feedback-forms/` → ignore (project content); `feedback/archive/` → consider README only; `feedback/archive/YYYY-MM/` → ignore; `evaluation-reports/` and `evaluation-reports/archive/` → ignore content but archive scaffolding is structural; `proposals/` and `proposals/old/` → ignore content but presence is structural; `tools/linkWatcher/logs/` → ignore content, ensure dir exists; `tools/linkwatcher/.linkwatcher-ignore` → consider (skeleton). For state-tracking files specifically (`process-improvement-tracking.md`, `test-tracking.md`, `e2e-test-tracking.md`, `performance-test-tracking.md`, `feature-tracking.md`, etc.), explicit guidance: compare section headers AND table column shapes AND status-legend rows, not just file presence. This single document would have prevented all 3 catch-up rounds in this session.
2. Add `BlueprintSync` variant to `New-TempTaskState.ps1` with sections matching the actual sync workflow: Session Parameters / Per-Item Classification table / Notes on Specific Items / Session Log / Completion Criteria.
3. Add a "Known Protected Artifacts" subsection to the framework-blueprint-sync task definition under "Per-Directory Handling Rules" → "Root files", listing intentional cross-project items (currently: `ratings.db`, `ratings.db.bak-*`). Future syncs read this list before classifying.
4. Tighten the validation rule from "No project-specific identifier leaked" to "No project-specific identifier leaked, AND any populated DB/binary file at blueprint root has user confirmation as either protected or leak."
5. Add a "Reference Verification" sub-step to Step 8 (Classification) for registry prefixes / templates / scripts: when the blueprint has a prefix the project lacks, grep both trees' framework code for references before classifying as stale. DOC-7 worked correctly because I did this check. Bake it into the process so future agents do it automatically.
6. For agent-delegated structural comparison work, add a self-verification step in PF-TSK-087: when an agent reports drift in a doc-map / registry / state-file, verify by direct grep before applying. The PD-documentation-map.md false positive in this session would have been caught immediately by a 2-line grep of `^## ` headers on each side.
