## Improvement Suggestions

### What worked well
- **Step 5 lightweight-path criteria** as a clear gate to skip Phase 2/3/4 boilerplate.
- **Active-pushback discipline** at the Step 5 checkpoint — presenting three alternatives (closeout-discipline / lightweight tracking / full lifecycle) instead of executing the IMP as written. The user reframed to a much better proposal; the original IMP-686 framing was disproportionate.
- **PF-IMP ID pool reuse** for pilots — much cleaner than introducing a separate ID prefix or registry. "Pilot is an IMP with a multi-stage lifecycle" is a solid abstraction.
- **Existing-script extension over new-script creation** — backward-compatible additive parameters kept the framework's script count flat.
- **2-state lifecycle (Active / Resolved)** — earlier 3-state proposal (Active / Awaiting Decision / Resolved) was correctly simplified after challenging what action each state drives.

- **Pre-Concept Analysis depth scaling** — sub-steps (a, a2, b, c, d, e) feel mandatory regardless of extension complexity. For an obvious modification-type extension, a lighter Pre-Concept gate would reduce friction.
- **Modification template heaviness** — already captured as PF-IMP-682. The 4-phase 15-step Implementation Roadmap section in the modification template is too heavy for single-session lightweight extensions.

## Additional Context

### Task-specific challenges
The original IMP-686 framing ("design and implement a proper pilot lifecycle process" with `pilot-tracking.md`, registration script, observation/decision template, `ai-tasks.md` routing entry) was substantially heavier than the actual problem warranted. This is exactly the scenario the "Apply IMP Minimum Viability Rigorously" memory rule targets. The Pre-Concept Analysis surfaced that pilots have run only ONCE in the framework's history (the soak-self-verification pilot), making a full lifecycle infrastructure premature. Through Step 5 active pushback and the user's reframing across multiple rounds, the proposal collapsed to: 1 task definition modification + 1 state file section + 2 existing-script extensions + 1 infrastructure doc note. ~10x scope reduction from the IMP's literal framing.

### Integration with other tools
- **Active Pilots ↔ script-soak-tracking.md**: pilot rows reference soak rows by Script ID (Adopters column). No automated cross-reference, but operators can navigate manually.
- **Active Pilots ↔ Update-ProcessImprovement.ps1**: same script handles regular IMP transitions and pilot Active/Resolved transitions, with location-based dispatch in Main. Single tool, two responsibilities.
- **Concept doc auto-archive**: when a pilot is resolved, Update-ProcessImprovement.ps1 looks up the concept file by frontmatter `id:` scan in `proposals/`, then moves it to `proposals/old/`. LinkWatcher then auto-updates references. This is the cleanest auto-archive the framework has.
