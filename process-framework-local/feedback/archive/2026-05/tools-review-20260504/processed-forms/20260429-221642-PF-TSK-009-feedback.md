## Improvement Suggestions

### What worked well
- Step 2 evaluation table caught minimum-viability on all 3 IMPs (every proposed change WAS the minimum viable version — no scope creep)
- Step 5 checkpoint format (Problem Summary → Eval table → Approach + Pushback options) gave the human partner clean redirect points; the IMP-623 scope-expansion question and IMP-628/IMP-632 scope-restriction questions all resolved cleanly
- Step 11 grep sweep was decisive on IMP-632 (zero external surfaces; self-contained fix confirmed) and surfaced adjacent fixes on IMP-623 (template + state template)
- Mid-session dogfooding: IMP-623's convention doc fixed canonical-ID lookup for IMP-628 and IMP-632; IMP-628's bash-quoting fix was applied to its own completion call (and IMP-632's), with verified backtick preservation

### What could be improved
- **Default `pwsh.exe -File path.ps1 -?` does NOT show .PARAMETER blocks** (only SYNOPSIS + SYNTAX + DESCRIPTION). Agents must use `pwsh.exe -Command 'Get-Help path.ps1 -Detailed'` or `-Parameter <Name>` to see parameter notes. The IMP-628 BASH GOTCHA warning lives in `.PARAMETER ValidationNotes` — it's invisible at default `-?`. Two ways to address: (1) Document this in a script-helpers guide (or in CLAUDE.md's PowerShell section), so agents reflexively reach for `-Detailed` when reading parameter help; (2) Move critical agent-facing warnings into the .DESCRIPTION block where they ARE visible at default `-?`. (1) is lower-cost but relies on agent memory; (2) is more reliable but bloats DESCRIPTION.
- Steps 13 and 14 collapse for trivial doc-class IMPs (1-2 file edits with verified grep sweep) — Step 14 ends up re-confirming Step 13. Not worth a process change for low recurrence; flagging only.

### Specific suggestions
- Add a one-line note to PF-TSK-009 (or to `process-framework/guides/support/script-development-quick-reference.md` if it has a "Reading PowerShell help" section): "When reading PowerShell parameter help, use `Get-Help path.ps1 -Parameter <Name>` or `-Detailed`/`-Full`. Default `pwsh.exe -File path.ps1 -?` shows only SYNOPSIS/SYNTAX/DESCRIPTION — `.PARAMETER` blocks (where in-script gotchas like the Update-ProcessImprovement.ps1 BASH GOTCHA live) are not rendered."
- Consider tightening the New-FeedbackForm.ps1 post-creation banner from ~10 lines to 2-3 lines. The current emoji-heavy block has signal-to-noise issues — agents who skip 10 lines will skip 3, and agents who don't will read either. Low-priority.
