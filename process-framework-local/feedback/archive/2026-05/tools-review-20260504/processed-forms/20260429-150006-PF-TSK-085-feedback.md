## Improvement Suggestions

### What worked well
- **Audit gate verification via the audit report itself**, not the tracking column. Tracking-file Audit Status was stale (`🔄 Needs Update` referencing TE-TAR-070), but the actual current audit TE-TAR-071 was `✅ Audit Approved`. Reading the report directly is the right discipline.
- **3-run capture** (per audit's improvement opportunity) produced very clean baselines — PH-007 op-delta was identical (4.9MB) across all 3 runs, and PH-008 raw avg was within ±0.4% across runs (vs ±15.9% in the audit's earlier 3 runs).
- **Sub-ID convention from BM tests transferred cleanly** to PH-007/PH-008 multi-metric reporting (PH-007-net, PH-007-op-delta, PH-008-avg-raw, PH-008-peak).
- **Update-PerformanceTracking.ps1 + performance_db.py stay in sync** without explicit cross-references — the agent acts as the synchronization layer, which is fine given the volume of captures.

### What could be improved
- **Audit gate language in PF-TSK-085** says "Audit Status is empty or `⬜ Not Audited`" must pass audit first. But PH-007/PH-008 had `🔄 Needs Update` (a different non-Approved state, referencing the prior audit). Strictly speaking, the rule didn't trigger — but I treated it as triggering anyway and flagged to the user. The right rule is "anything other than `✅ Audit Approved` blocks", not just "empty or Not Audited".
- **Tracking file's Audit Status column lags the audit report itself.** TE-TAR-071 was approved 2026-04-29 but the tracking column still showed `🔄 Needs Update` (stale from TE-TAR-070). The audit's own action item flags this for `Update-TestFileAuditState.ps1`; that the action items hadn't been executed by the time PF-TSK-085 ran is a workflow handoff issue between PF-TSK-030 (Test Audit) and PF-TSK-085. Worth a stronger link from audit task → "before declaring audit complete, run `Update-TestFileAuditState.ps1`".
- **performance_db.py lacks a `list` subcommand** for discovering known test_ids. Had to use `export | head` to find the multi-metric sub-ID convention.
- **Multi-metric sub-ID convention is undocumented.** It's clearly the established pattern from PH-001..PH-005 entries, but a guide line in performance-testing-guide.md or the performance_db.py help text would prevent ad-hoc divergence.
- **Update-PerformanceTracking.ps1 Summary log line** shows new total state, not delta from this run. With other rows changing in parallel (parallel session, etc.), this can momentarily look like the script over-reached.

### Specific suggestions
1. **PF-TSK-085 Step 2 audit gate**: Tighten to "Audit Status must be `✅ Audit Approved`. Any other value (empty, `⬜ Not Audited`, `🔄 Needs Update`, `🔴 Audit Failed`) blocks baseline capture."
2. **PF-TSK-030 finalization step**: Add explicit "Run `Update-TestFileAuditState.ps1 -TestType Performance -AuditStatus 'Audit Approved'`" as a numbered finalization step so the tracking-file Audit Status column never lags the audit report.
3. **performance_db.py**: Add `list [--prefix BM]` subcommand. Add a `--summary` flag to `trend` for mean/min/max over the last N entries.
4. **performance-testing-guide.md**: Document the multi-metric sub-ID convention (`<TestID>-<metric>`) with a note that the primary asserted metric is the canonical baseline.
5. **Update-PerformanceTracking.ps1**: Change Summary log line to `Updated row: PH-007. Current totals: ...` so a single-row update doesn't read as a bulk action.

## Additional Context

### Task-specific challenges
- **Stale audit status in tracking file**: PH-007/PH-008 rows had Audit Status `🔄 Needs Update` referencing the prior audit (TE-TAR-070), but the current re-audit (TE-TAR-071, same day) was `✅ Audit Approved`. Resolved by reading the audit report directly. This was an artifact of the audit task's `Update-TestFileAuditState.ps1` action item not yet being executed.
- **Parallel session state changes**: The tracking file's BM rows and `📋 Needs Baseline` → other states had been updated by a parallel session between my initial Read and my Update calls. Caused a momentary "did the script over-reach?" question that resolved on inspection of git diff and Summary recalculation logic.
- **Test output truncation on first run**: The first single-execution `pytest -v -s` produced enough log output that PH-007's measurements scrolled off the visible window. Worked around by re-running with `grep` filter capturing only the print-statement lines.
- **Multi-metric DB recording convention**: PH-007 has 2 useful metrics (net memory, op delta), PH-008 has 2 (raw avg, peak). The DB doesn't natively handle this; convention is sub-IDs (`PH-007-net`, `PH-007-op-delta`). Convention was discoverable from `export` output but not documented.

### Integration with other tools
- **pytest** (test runner, external): Used to execute `TestPerformanceMetrics::test_memory_usage_monitoring` and `::test_cpu_usage_monitoring`. Output verbosity (`-v -s`) was both the strength (full diagnostics) and the issue (truncation in tool windows).
- **TE-TAR-071 audit report** (output of PF-TSK-030): Source of truth for audit status when tracking column was stale. The reference link from PF-TSK-085 to "verify Audit Status" worked; the implicit dependency on the audit task's tracking-update action item is the weak point.
- **Update-PerformanceTracking.ps1 ↔ performance-test-tracking.md**: Clean integration; script handles row update + Summary recalc atomically.
- **performance_db.py ↔ git**: Auto-captures git commit hash with each measurement — useful for future regression diagnosis.
