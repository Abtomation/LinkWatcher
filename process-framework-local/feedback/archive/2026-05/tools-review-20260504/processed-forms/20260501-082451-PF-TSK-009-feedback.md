## Improvement Suggestions

### What worked well
- Step 3 evaluation table with the conciseness rule (one-line summary when all favorable, full table when concerns surface) — kept IMP-652 checkpoint compact while giving IMP-653/-654 room for proper analysis.
- Minimum-Viability gate at Step 6 — directly led to Option B for IMP-654 (verification-preserving grep recipe instead of the IMP-proposed batched-N/A shortcut, which had a weaker premise than the existing 3 shortcuts).
- Domain heuristic in code-refactoring-task.md L94 — cleanly grounded the IMP-653 rejection rationale; user invoked it without having to articulate the principle from scratch.
- Update-ProcessImprovement.ps1 status auto-defaults (Impact = "—" for Rejected) — small but reduces friction.
- Step 11 silent housekeeping — caught the parallel inconsistency in process-framework-task-registry.md without surfacing noise.

### What could be improved
- IMP-652 evaluation initially defaulted Framework Fit to "Good" without rigorously checking whether the proposed change conflicted with cross-task routing rules. The user had to surface the framework-routing concern for IMP-653 (the IMP would have normalized misuse of PF-TSK-022 for `process-framework/` work). The IMP-652 evaluation didn't have the same problem because the change was self-contained, but the same lookup discipline should have been applied.
- The L8 shortcuts section in code-refactoring-lightweight-path.md is approaching scannability limits — 3 (now 4) callouts plus a 9-item list is borderline. Future shortcut additions should consider sub-organizing.

### Specific suggestions
- Step 3 Framework Fit row could explicitly prompt: "If the IMP proposes adding a workflow shortcut or guidance, check whether the work it would enable is already routed to a different task by the parent task's reclassification block." Catches IMP-653-style misroutes before reaching the human checkpoint.
- For IMPs proposing additions to growing checklists/shortcut sections (like L8 here), the evaluation table could include a "Section Crowding" check — "is the host section already approaching the limit where adding more items degrades scannability?" This would surface refactoring/sub-organizing as an alternative to indefinite addition.
