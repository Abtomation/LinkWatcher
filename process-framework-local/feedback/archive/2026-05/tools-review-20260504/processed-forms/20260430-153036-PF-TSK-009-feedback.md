## Improvement Suggestions

### What worked well
- Step 6 checkpoint format (Problem Summary → Evaluation table → Proposed approach) consistently triggered the right kind of human input. For IMP-667 it surfaced the band-aid framing for pushback before commitment; for IMP-672 it isolated the `Last Run` scope decision for explicit human judgment.
- Step 3 conciseness rule (one-line summary when all favorable) prevented bureaucratic table-rendering on simple IMPs.
- Update-ProcessImprovement.ps1's auto-managed Update History entries removed manual boilerplate.
- feedback_db.py's `--batch` mode for multi-tool change logging.
- The `git log -S "..."` pattern for finding when an artifact was introduced and what its initial purpose was — applied to IMP-672 (Last Executed column) and conceptually to IMP-667 (audit tolerance origin).

### What could be improved
- Update-ProcessImprovement.ps1 lacks `-EditDescription` / `-EditNotes` operation modes (parallel to PF-IMP-657 for Update-TechDebt.ps1). Manual tracking-file edits are error-prone (column-alignment risk).
- PF-TSK-009 Step 2 ("Verify the problem") should explicitly mention the "check initial purpose via git history" technique. Currently the methods listed ("grep recent sessions for the symptom, read the artifact under discussion, check IMP history, inspect the feedback DB") don't include git history of the artifact itself.
- The minimum-viability gate in Step 3 worked, but it triggered only because the human asked the right question on IMP-667. The framework relies on human judgment to spot when an IMP description is itself a band-aid framing — could there be a more explicit prompt? E.g., a sub-question under Minimum Viability: "Does the IMP description describe a workaround for a deeper structural issue? If yes, propose the structural alternative explicitly."

### Specific suggestions
1. **Extend Update-ProcessImprovement.ps1** with `-EditDescription <new>` and `-EditNotes <new>` operation modes (mirror PF-IMP-657 design). Same atomicity as the resolve path.
2. **Add to PF-TSK-009 Step 2 methods** (verify the problem): "When the change involves removing or modifying an existing artifact (field, column, function, parameter), check its initial purpose via `git log -S 'artifact-name' --reverse` and the originating commit's diff. Current usage may have drifted from documented intent — make decisions against original purpose, not just current state."
3. **Consider for PF-TSK-009 Step 3 Minimum Viability gate**: Add an explicit "structural alternative" prompt — when an IMP describes adding documentation/guidance/workaround, ask whether the underlying artifact could be redesigned to make the workaround unnecessary.
