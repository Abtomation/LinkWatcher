## Improvement Suggestions

### What worked well
- **Batch IMP registration via `-BatchFile`**: 38 IMPs in one command vs. 38 individual invocations. Single most efficient step in the entire workflow.
- **Step 11 review-summary-skeleton-first sequencing**: gave a stable `-SourceLink` filename for every IMP without circular dependency.
- **Routing Decision table at Step 12**: classification (IMP/Bug/Tech Debt/Feature Request) was unambiguous; all 38 items routed to IMP cleanly.
- **Deduplication callout at Step 12**: caught PF-IMP-620, PF-IMP-584, and PF-IMP-223 as existing — prevented 3 duplicate registrations.
- **"Each distinct actionable change = 1 IMP" granularity rule**: produced 38 distinct entries rather than collapsing into a few mega-entries.
- **Per-task-group analysis structure in the review-summary template**: rolling up ratings + themes + IMP IDs per group made the summary scannable for trend analysis.

### What could be improved
- **Form-archival recovery procedure missing AND parallel-session coordination missing**: the task definition correctly enforces strict archival rules but doesn't describe (a) what to do when the wrong forms get moved or (b) how parallel Tools Review sessions on the same date should coordinate. My initial `mv` command's archive folder ended up with 70+ forms instead of the named 34 — most plausibly because session 1 of the Tools Review finished concurrently and dropped its own analyzed forms into the same `tools-review-YYYYMMDD/processed-forms/` directory (the dated archive folder is shared by all same-date sessions). I treated the unexpected forms as misplacement and restored 40 to active — but if session 1 had legitimately archived them, my restore was the error. Two needed additions: (1) "if you discover wrong forms in the archive, restore them via `mv` from archive back to feedback-forms" recovery note, AND (2) parallel-session guidance — perhaps suffix the archive folder with a session number (`tools-review-YYYYMMDD-sessionN/`) so sessions don't clobber each other's archive folders, or require a coordination check at Step 14 before restoring "unexpected" forms.
- **Step 5 vs Step 10 checkpoint distinction is fuzzy in practice**: when the prep checkpoint (Step 5) presents inventory + initial themes thoroughly, the user's "continue" can implicitly approve the analysis content as well, collapsing Steps 5 and 10. The task could clarify whether collapse is acceptable when prep analysis is comprehensive, or insist on the second checkpoint being a distinct gate.
- **No guidance on broader-sweep IMP registration** (the "Observe Narrow Fix Before Broadening" pattern lives in agent memory but not in the task definition): when a session-2 form suggests applying an existing IMP's fix to another script (e.g., trim the verbose banner that PF-IMP-584 fixes for one script, to other scripts too), the right call is to observe rather than file a separate IMP — but the task doesn't say so. A one-line guidance would normalize the practice.

### Specific suggestions
- Add a "Form-Archival Recovery + Parallel-Session Coordination" sub-step: (a) "If the wrong forms were moved during archival, restore via `mv` from `archive/.../processed-forms/` back to `feedback-forms/`. Verify task-group counts match the session scope before recording ratings." (b) "If multiple Tools Review sessions run on the same date, suffix the archive folder with a session number (`tools-review-YYYYMMDD-sessionN/`) to prevent clobbering. Before restoring 'unexpected' forms from a shared dated archive, check whether a parallel session may have correctly archived them."
- Clarify in Step 10 whether the execution checkpoint can be implicitly approved by the same "continue" that approves the prep checkpoint, or whether a distinct human acknowledgment is required.
- Add a "Broader-Sweep Observation" guideline to Step 12's deduplication callout: "If a feedback item suggests applying an existing IMP's fix pattern to additional artifacts (e.g., extending PF-IMP-XXX's script-specific change to other scripts), record this as an observation in the review summary rather than registering a new IMP, unless the broader sweep is itself the actionable change."

## Additional Context

### Task-specific challenges
- **Over-archive incident**: My initial archive `mv` command (a single bash command listing 34 specific filenames + a destination directory) somehow moved 70+ forms to the archive instead of the named 34. Forms from PF-TSK-005, -007, -009, -014, -026, -030, -041 (not in session 2 scope) ended up in the archive. Root cause unclear — possible explanations: (a) **session 1 of the Tools Review finishing concurrently** and archiving its own forms into the same `tools-review-20260429/processed-forms/` directory (likely — the dated archive folder is shared by all sessions on the same date, so any parallel Tools Review session would write into the same location); (b) bash line-continuation issue in my long `mv` command; (c) other parallel session interference (two new TSK-022 forms appeared in `feedback-forms/` during my session at 14:05 and 14:08 from another agent). The session-1-overlap hypothesis is the most plausible because the over-archived forms are exactly the kind of mix (TSK-005/007/009/014/026/030/041) that another Tools Review session would naturally process. I identified the 40 over-archived forms by listing the archive vs. my session 2 list, and restored them via `mv` back to `feedback-forms/`. Final state: archive has 34 forms (27+2+2+1+1+1 by task), active folder retains all unrelated forms. **If session 1 was indeed concurrent, the restore was a mistake** — those forms may have been correctly archived by session 1. Worth flagging as a process error worth (a) a recovery procedure for genuinely-misplaced cases AND (b) coordination guidance for parallel Tools Review sessions sharing the same dated archive directory (captured under What Could Be Improved).
- **Context window pressure**: 34 forms × ~2-5KB each = significant context consumption during sequential reading. The task definition's "Analysis quality over speed" rule (don't parallelize form analysis) was followed — sequential reading caught patterns batch scanning would miss. Reading was paced in 6-form batches to balance throughput with care.

### Integration with other tools
- **`New-ReviewSummary.ps1` ↔ `New-ProcessImprovement.ps1 -BatchFile`**: stable-filename handoff via `-SourceLink`.
- **`New-ProcessImprovement.ps1 -BatchFile` ↔ ID registry**: 38 sequential IDs (PF-IMP-647 through PF-IMP-684) assigned without gaps; registry counter incremented atomically.
- **`extract_ratings.py` ↔ `feedback_db.py record`**: pipeline (markdown → JSON → SQLite) runs end-to-end with no manual intermediate edits. The `--archived-prefix` parameter ensures DB records reference forms at their final post-archive paths.
