## Improvement Suggestions

### What worked well
- **Step 2 evaluation table as quality gate**: Caught a wrong-rationale IMP (645) when human partner challenged the source-form premise. The "Still Valid?" criterion gave the framework leverage for a clean pivot rather than barreling forward with imported wording.
- **Step 11 grep sweep + Step 13 hit list**: On IMP-645 the sweep returned 11 files referencing context-window concepts; the hit list discipline cleanly separated the 4 in-scope edits from the 9 observed-but-deferred files. On IMP-639 the sweep similarly catalogued ~12 active framework references vs. ~60 historical artifacts.
- **Live-test snapshot/restore pattern (IMP-639)**: For script changes that mutate tracked state, copy → run → diff → restore is a safe verification path within the no-stash/no-checkout constraint.
- **Per-IMP claim/complete via Update-ProcessImprovement.ps1**: The script correctly handled "InProgress in Current section" → "Completed moved to Completed section" without manual table moves.
- **3-IMP session limit**: Acted as a natural session close; quality didn't degrade through IMP-645 even though that IMP grew unexpectedly (from one-line callout to 5-file cleanup after the rationale challenge).

### What could be improved
- **Step 2 doesn't probe rationale verifiability**: The "Still Valid?" criterion asks whether the *problem* is still present, but not whether the *rationale* is independently verifiable by the agent. On IMP-645 I echoed "context window run hot" from the source feedback form without flagging it as something I can't introspect on. (Single-occurrence; not proposing a callout from one incident, per existing memory rule.)
- **Live-test restoration relies on shell-side discipline**: The snapshot-and-restore approach for IMP-639 was effective but ad-hoc. There's no script-level support for "rehearse a state file mutation in a temp copy". This is borderline within scope of the script under modification (Update-TestFileAuditState.ps1), not the task definition.

### Specific suggestions
- *(Single-occurrence; observation only — no doc change proposed)*: Consider whether Step 2 should add a sub-prompt like "Does the rationale rest on agent-internal state (context size, attention, memory) that the agent can't independently verify? If yes, reframe in observable-behavior terms before proceeding." Defer until a second occurrence justifies the callout.
