## Improvement Suggestions

### What worked well

- **Structured Step 3 evaluation table** drove the rejection of IMP-715 directly via the Minimum Viability and Root-Cause Targeting gates. Without those gates I likely would have implemented a 7-task Step 0 addition that wouldn't have addressed the actual stated problem.
- **Step 6 "1-3 surviving options" format** produced a tractable choice surface for IMP-715 (reject / one-time audit / Framework Evaluation tweak). User picked B+C; combined outcome was strictly better than the original IMP.
- **Compressed/Skip variants of Step 13** (compressed format already existed; Skip option added this session at user request) appropriately reduce ceremony for low-risk plan-matching changes. Used Skip for both IMP-711 and IMP-716 finalization.
- **"Always use scripts" discipline** caught what would have been a manual-edit drift in process-improvement-tracking.md — Update-ProcessImprovement.ps1 handled summary-count updates and history rows automatically.

### What could be improved

- **"Rejected with replacement" pattern is undocumented**. When IMP-715 evaluation produced 2 substitutions (new IMP + inline edit), I had to synthesize that this counts as a valid "Rejected" outcome. The rejection rationale text is the only place to capture the substitution links.
- **Step 17 trigger conflates "session done" with "IMP done"**. IMP-716 setup-only this session triggered Step 17 (final feedback form) the same as if it had been fully implemented. Worth clarifying that Step 17 closes the session regardless of the per-IMP state of the session's IMPs.
- **Step 10 risk classification edges are slightly fuzzy** — e.g., "additive callout" appears in low-risk; "behavior changes within one task" in medium-risk. Where does a Skip-option add to a checkpoint sit? I called it low-risk because it relaxes process, doesn't change semantics, but the reasoning was non-trivial.
- **temp-process-improvement-state-template.md is verbose** for the volume of content most state files actually need. Many placeholders resolve to "None" or "N/A". An optional-section model could halve the boilerplate.
- **No automated link from IMP row to its temp state file**. When a multi-session IMP gets a temp state file (e.g., IMP-716 → PF-STA-101), the IMP's tracking row has no Tracking column or Notes-column link to the state file. Discovery requires either reading the Update History (where the state file ID appears as plain text) or knowing to search `process-framework-local/state-tracking/temporary/`. Surfaced when human asked "where is the progress of this IMP stored?" — the answer required cross-referencing two files. Manually patched the IMP-716 row to add the link, but the root cause is that neither (a) `New-TempTaskState.ps1` (ProcessImprovement variant) auto-posts a Tracking link to the linked IMP, nor (b) `Update-ProcessImprovement.ps1` has an `-EditNotes` option, nor (c) the convention is documented in PF-TSK-009 Step 7.

### Specific suggestions

- Add a sentence to Step 14 acknowledging "Rejected with replacement IMPs/inline edits" as a valid outcome: *"If the rejection produces substitution work (new IMPs, inline edits to existing tools), capture the substitution IDs in -ValidationNotes so the rejection rationale links forward."*
- Add a sub-bullet to Step 17 clarifying that it closes the session, not the per-IMP state: *"Trigger this regardless of per-IMP state — completed, rejected, deferred, or setup-only multi-session IMPs all count toward the session limit, and the feedback form covers all of them."*
- Consider a "low/medium/high" tiebreaker line in Step 10: *"If unsure between two adjacent classes, pick the lower if the change matches the Step 6 plan exactly with no human-facing UX impact, else the higher."*
- Slim down temp-process-improvement-state-template.md by making Phase boilerplate, Issues/Blockers, and per-session subsections optional rather than always-present.
- Wire `New-TempTaskState.ps1 -Variant ProcessImprovement` to accept an `-ImprovementId` parameter; on creation, automatically post a Tracking link to the IMP's Notes column (or the row's Update History) via `Update-ProcessImprovement.ps1`. Removes the manual cross-reference step and prevents the discovery gap surfaced this session. Alternatively, document the convention in PF-TSK-009 Step 7 if automation is too heavy.
