## Improvement Suggestions

### What worked well

- **Heuristic + out-of-scope list defined upfront** (in Session 1's setup) made per-occurrence decisions in Sessions A-D consistent and fast. The "When in doubt → retain" guidance with a documented rationale was the single most valuable tool in the audit.
- **Step 10's risk-class branching** correctly identified the bulk sweep as low-risk and eligible for batch execution + grep verification.
- **Step 13 skip-option** for low-risk + no-deviation changes prevented unnecessary checkpoint friction.
- **Pattern recognition during Sessions A and B** drove the Session B → Session C scope expansion (canonical templated pair sweep) — turning what would have been weeks of per-file audit into a single bulk operation.
- **Verification grep loop** (broader doubled-emoji pattern, then load-bearing keyword retention check) caught 3 additional Category 1+2 instances in concept templates that the original scope didn't anticipate.

### What could be improved

- **No documented bulk-logging pattern** for cases where one mechanical change touches dozens of artifacts. The canonical "task ID per affected task" convention requires per-task log entries; for the 55-task sweep this needed a custom script to build the batch JSON. A documented umbrella-entry pattern (or an accepted variant) would be cleaner.
- **Cadence guidance vs. bulk-treatment guidance is in tension**. The state file's "3-5 files per session" cadence presumed deliberative per-file work; Step 10's bulk endorsement allows much higher per-session volume. The user partner explicitly overrode the cadence to fit the entire audit into one conversation, which worked, but the framework documentation could acknowledge this tension explicitly.
- **The feedback form's own opening callout** (`🚨 CRITICAL COMPLETION REMINDER`) is a residual instance of the same pattern this IMP audited. Self-reflective scope-creep is out of bounds for IMP-716 (template wasn't in scanned scope), but it's a meta-observation worth noting.

### Specific suggestions

- **Document a bulk-sweep logging pattern** for Step 12: when one mechanical change touches N artifacts following an identical heuristic, allow a single umbrella entry against the orchestrating task ID with a description listing affected artifact IDs/categories. Currently must be inferred.
- **Add an "Audit-style task" example** to the Process Improvement task definition or its implementation guide showing how Sessions A-N + Phase 3 finalization compose for multi-session audits beyond the simple per-IMP execution model. Audits don't fit neatly into "select an improvement, evaluate, execute, mark complete" — they iterate within a single IMP across many sessions.
- **Consider an `--imp-bulk` flag for feedback_db.py log-change** that takes a list of artifact IDs and a single description, generating the batch JSON internally.
