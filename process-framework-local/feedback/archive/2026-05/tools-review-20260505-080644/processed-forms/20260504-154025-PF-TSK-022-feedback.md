## Improvement Suggestions

### What worked well
- **PF-IMP-693 precedent as a pattern source**. Sister fix (same defect class, completed days earlier) gave me a battle-tested helper-substitution shape to mirror — `ConvertFrom-MarkdownTable -Section X -IncludeRawLine` + `Where-Object { $_.ID -eq $TargetId }` + `Split-MarkdownTableRow $row._RawLine`. No re-derivation cost.
- **L8 grep-recipe shortcut**. Single `rg -l <identifier>` across 8 doc surfaces produced explicit evidence (0 matches) backing one collective N/A justification. Concrete substitute for 8 hand-written justification lines.
- **L5/L10 checkpoints at the right granularity**. L5 caught the L4 PowerShell-vs-Python structural gap before any code was touched. L10 gated state-file writes (which were irreversible by L11) behind explicit approval.
- **Defect-class proof via synthetic input**. The plan's "Manual Functional Verification" section forced me to construct a cross-column-pollution test case — which surfaced that the OLD regex returned a malformed substring in Test 1 (mixing PD-FRQ-001's ID-text from row 1's Source column with row 1's date). Concrete proof the refactor closes the defect class, not just speculation.

### What could be improved
- **L4's Python/pytest assumption breaks for PowerShell framework tooling refactors**. `New-TestFile.ps1` generates Python pytest files; project has no Pester infrastructure. Second occurrence (PF-IMP-693 hit the same gap days earlier).
- **L11 doesn't acknowledge IMP routing**. Bullet says `Update-TechDebt.ps1` but PF-TSK-022 Step 1 explicitly accepts IMPs via the domain heuristic. The IMP closure tool is `Update-ProcessImprovement.ps1` which has different parameters (`-Impact`, `-ValidationNotes` instead of `-DebtId`, `-ResolutionNotes`).
- **Update-ProcessImprovement.ps1 default `-UpdatedBy` attribution**. Defaults to "AI Agent (PF-TSK-009)" (Process Improvement) but Code Refactoring (PF-TSK-022) is a legitimate caller. Caller should override; Lightweight Path L11 doesn't note this.

### Specific suggestions
- **Add a "framework-tooling-only" exemption to L4** (similar to existing documentation-only and build-config-only exemptions): "If the refactoring targets PowerShell framework tooling under `process-framework/scripts/` and the project has no Pester infrastructure, skip the New-TestFile.ps1 step. Verification strategy: manual functional invocation against real state files — happy path with valid ID, error path with non-existent ID, and a defect-specific synthetic case (e.g., cross-column pollution for row-lookup refactors). Record the verification recipe and results in the plan's Test Coverage Assessment field."
- **Update L11's first bullet** to add an IMP-routing path: "If the resolved item is a TD: `Update-TechDebt.ps1 -DebtId 'TD###' -NewStatus 'Resolved' -ResolutionNotes '...'`. If the resolved item is an IMP (per PF-TSK-022 Step 1 domain heuristic): `Update-ProcessImprovement.ps1 -ImprovementId 'PF-IMP-NNN' -NewStatus 'Completed' -Impact '<H/M/L>' -ValidationNotes '...' -UpdatedBy 'AI Agent (PF-TSK-022)'`."

## Additional Context

### Task-specific challenges
- **No Pester infrastructure for PowerShell framework tooling**. L4 instructs `New-TestFile.ps1 -TestType "unit"` to write characterization tests; that script generates Python pytest files only. Worked around with manual functional verification, mirroring the PF-IMP-693 precedent. Surfaced and got explicit approval at L5 (option A).
- **Defect class is latent, not active** — proving the refactor closes it required constructing a synthetic input where the target ID appears as a full-cell value in a non-ID column. My first synthetic attempt (Test 1 v1) used "Migrated from PD-FRQ-001" as the cross-column text; the OLD regex correctly skipped that row because `\s*` won't match through "Migrated from ". I had to redesign with the ID as a clean full-cell value to actually trigger the malformed match the OLD regex produces.

### Integration with other tools
- **PF-IMP-693 sister fix in `Update-ProcessImprovement.ps1`** provided the template for both the helper-substitution shape and the closure flow (`Update-ProcessImprovement.ps1` itself uses `ConvertFrom-MarkdownTable` exactly because of IMP-693). PF-IMP-694 was a near-mechanical replay.
- **TableOperations.psm1** v2.0 helpers are well-suited to this pattern — `ConvertFrom-MarkdownTable -IncludeRawLine` returns objects with both column-keyed properties (for safe lookup via `Where-Object`) and a `_RawLine` property (for the existing `$Content.Replace($currentEntry, $updatedEntry)` semantics that the legacy code relied on). Drop-in replacement.
- **Run-Tests.ps1 `-All`** is fast enough (~110s) that running the L3 baseline and L7 regression tests in series is cheap. Test diff was identical (expected, since no Python product code touched).
