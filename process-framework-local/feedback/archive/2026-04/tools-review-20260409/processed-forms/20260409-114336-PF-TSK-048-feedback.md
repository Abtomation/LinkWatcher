## Improvement Suggestions

### Self-Analysis: What the Human Partner Had to Correct

Tracing through the conversation, the human partner made 7 significant challenges. Each one reveals a gap in my approach:

**1. "Did you consider on what abstraction level we need performance tests?"**
- **What I did wrong**: Jumped straight to implementation artifacts (guide, baselines, template) without first establishing the conceptual model. Copied PF-EVR-010's industry categories (microbenchmark/load/stress/endurance) verbatim.
- **What I should have done**: Before any artifact design, ask: "What are the natural abstraction levels in LinkWatcher's architecture? How do they map to performance testing?" Read the existing test files first to see what levels they actually test at.
- **Root cause**: Started implementation-focused instead of conceptual. Treated PF-EVR-010 as a specification to execute rather than a source analysis to interpret.

**2. "I am challenging the assignment to the feature level"**
- **What I did wrong**: Initially tried to map performance tests to features despite evidence against it. The existing code already marks performance tests as `feature("cross-cutting")` — the answer was in the code I had already read.
- **What I should have done**: When I read `test_benchmark.py` (line 25: `feature("cross-cutting")`), I should have recognized this as a signal that performance tests don't fit feature-based tracking.
- **Root cause**: Didn't connect evidence from code reading to design decisions. Read the files but didn't think about what they were telling me.

**3. "I wonder if we should use a separate tracking file"**
- **What I did wrong**: Tried to force performance tests into test-tracking.md (adding Feature ID to orphaned rows) instead of recognizing the fundamental mismatch: test-tracking.md is feature-organized, performance tests are cross-cutting.
- **What I should have done**: Study the existing tracking files first. e2e-test-tracking.md already exists as a precedent for test types with distinct lifecycles getting their own tracking file. The pattern was already established.
- **Root cause**: Didn't study existing project patterns before designing new ones. Assumed existing structures should be reused rather than evaluating fit.

**4. "Do not hesitate to suggest new tasks"**
- **What I did wrong**: Concept v1 said "No new task definitions — artifacts consumed by existing tasks." I was being conservative, trying to avoid scope expansion, rather than honestly evaluating whether existing tasks can handle performance testing.
- **What I should have done**: Ask "Does PF-TSK-053 genuinely support performance test creation methodology?" When I later read PF-TSK-053 in detail, the answer was clearly no — it has one vague sentence about PE. I should have read it before claiming existing tasks were sufficient.
- **Root cause**: Optimized for minimal change rather than correct design. Avoiding new tasks isn't a virtue if existing tasks don't fit.

**5. "Who is responsible to plan which performance tests need to be created?"**
- **What I did wrong**: Concept defined "Performance Test Creation" task but didn't specify who decides *what* to test. The planning gap was invisible to me until asked.
- **What I should have done**: Trace the full lifecycle end-to-end before proposing components: who plans → who creates → who runs → who records → who reviews. For functional tests this chain is: TDD → TSK-033 → TSK-053 → TSK-030. I should have built the equivalent chain for performance tests.
- **Root cause**: Designed components in isolation rather than tracing the full workflow. Didn't ask "and then what happens?" at each step.

**6. "How is it with the e2e tests?" / "I fear the Performance Test Creation process doesn't know when to trigger"**
- **What I did wrong**: Didn't study the E2E precedent before designing the performance test lifecycle. E2E tests already solved the exact problems I was facing: separate tracking, lifecycle statuses (⬜ Not Created), multi-session visibility. The answer was in the project.
- **What I should have done**: Before designing ANY new tracking structure, read e2e-test-tracking.md to understand existing patterns. Ask: "How does the project already handle test types that don't fit the standard model?"
- **Root cause**: Designed from first principles instead of studying existing precedents. The project already solved this problem — I just didn't look.

**7. "Do we even need to mention the performance testing workflow to TSK-053?" / "Related Features column"**
- **What I did wrong**: Over-coupled PF-TSK-053 to the performance workflow (routing, triggering). Also designed the tracking file without thinking about practical usage (how do you know which tests to run after a code change?).
- **What I should have done**: Think from the user's perspective: "I just changed parser code. What do I do?" That immediately reveals the need for a feature-to-test mapping. And: "If PE is fully handled elsewhere, what does TSK-053 actually need to know?" — nothing.
- **Root cause**: Designed for architecture correctness rather than practical usage. Didn't adopt the developer's daily workflow perspective.

### What Would Have Prevented These Mistakes

1. **Study existing patterns first**. Before designing new tracking/lifecycle/tasks, read how the project already handles similar cases (E2E precedent). This would have prevented mistakes 3, 6.

2. **Start conceptual, not implementational**. Before listing artifacts, establish: What are the abstraction levels? What lifecycle do these tests follow? Who owns each step? This would have prevented mistakes 1, 2.

3. **Trace the full lifecycle end-to-end**. For every new workflow, walk through: who triggers → who plans → who creates → who runs → who records → who reviews → how do you know what's left? This would have prevented mistakes 5, 6.

4. **Evaluate fit honestly**. When proposing "use existing task X," actually read task X and ask "does this genuinely support the new requirement, or am I avoiding work?" This would have prevented mistake 4.

5. **Think from the user's perspective**. Ask "I just changed code in module X — what do I do?" and "I'm starting a new session — how do I see what's left?" This would have prevented mistakes 6, 7.

6. **Don't optimize for minimal change**. The goal is correct design, not minimal diff. New tasks and new files are fine when the existing framework doesn't fit. This would have prevented mistake 4.

## Human User Feedback

The human partner identified structured checklists that should guide concept development for framework extensions. These are not exhaustive but address the systematic gaps exposed in this session:

**General considerations for new concepts:**
- Scalability — will this work as the project grows?
- Abstraction level — at what level does this operate, and does it match the project's architecture?
- Ownership — who owns each artifact, process, and decision?

**For every new document/artifact:**
- Where does it live?
- Where is it referenced?
- Who creates it?
- Who is responsible for updating it?

**For every new task:**
- What is its trigger?
- Where is its output stored?
- How is the interface to the previous and next task ensured in a self-documenting way?
- What steps of other tasks are now redundant?

**Mandatory pre-concept reading:** The [Task Transition Guide](../../guides/framework/task-transition-guide.md) must be consulted before designing any extension to understand how the current framework's tasks connect and hand over to each other. This would have surfaced the PF-TSK-033 planning responsibility and PF-TSK-053 scope questions early.

Every major correction from the human partner was conceptual — abstraction levels, lifecycle ownership, existing precedents, practical usage — while I was focused on listing files and templates.

The 5 corrective principles to internalize for future extension work:
1. Study existing project patterns before designing new ones
2. Establish the conceptual model before listing artifacts
3. Trace the full lifecycle end-to-end before proposing components
4. Evaluate fit honestly — don't shoehorn into existing tasks
5. Think from the daily-usage perspective, not just architecture
