## Improvement Suggestions

### What worked well
- The E2E test framework clearly showed the failure (expected vs actual line-by-line comparison)
- Update-BugStatus.ps1 automated the full lifecycle cleanly
- Log analysis with structured JSON logs made root cause diagnosis straightforward
- The deferral mechanism in handler.py works correctly — no core code changes needed

### What could be improved
- The E2E orchestrator wastes ~30s starting its own LW for test cases that manage their own LW lifecycle (like TE-E2E-012)
- `start_linkwatcher_background.ps1` silently ignores unknown parameters — could use a param block to catch misuse

### Specific suggestions
- Consider adding a `skip_lw_start: true` frontmatter option to test-case.md for tests that manage their own LW lifecycle
- Add a param block to `start_linkwatcher_background.ps1` so unknown parameters like `-ProjectRoot` cause an error instead of being silently ignored

## Additional Context

### Task-specific challenges
- The core challenge was that the bug was actually in the E2E test infrastructure (run.ps1), not in LinkWatcher code. The previous fix (deferral mechanism) was correct but couldn't be verified because the test was broken.
- Significant iteration was needed to find the right number of ballast fixture files — too few and the scan finishes before the move, too many and the scan exceeds the wait timeout. Also discovered that `Start-Process` runs Python ~2.4x slower than bash background processes, requiring further adjustments.

### Integration with other tools
- Run-E2EAcceptanceTest.ps1 integrates well with Setup-TestEnvironment.ps1 and Verify-TestResult.ps1
- The bug tracking pipeline (Update-BugStatus.ps1) integrates cleanly with the task workflow
