## Improvement Suggestions

### What worked well
- Effort Assessment Gate's "independently verify" warning forced a real read of the integration narrative + grep of all production callers, which surfaced that the binary "gate or remove" framing in TD231's note was hiding a meaningful design decision.
- L3 baseline → L7 diff structure made "no new failures" verifiable rather than asserted.
- New-RefactoringPlan.ps1, Update-TechDebt.ps1, and the New-FeedbackForm.ps1 chain were friction-free.
- The "Resolved" pattern in the integration narrative (TD227, TD229, now TD231) is a great precedent — it keeps the dead-fields list shrinking visibly over time and provides a paper trail without bloating the live table.

### What could be improved
- **Run-Tests.ps1 swallows pytest summary output** — the wrapper completes with exit code 0 and "All tests completed successfully!" but doesn't show "XXX passed, X failed". L3 requires those numbers. Direct pytest invocation works around it but defeats the wrapper's purpose for this workflow.
- **Pre-existing test count drift is not attributable** — Validate-TestTracking flagged `test_logging.py: marker=29, actual=28` (and `test_validator.py: marker=75, actual=111`). Drift exists in both directions (test_query AST vs pytest collection vs test-tracking.md). Without per-row "last verified at commit X" metadata it's hard to attribute a warning to the current session vs prior drift; agents end up writing "I added 2, this is +2 from the last bad value" reasoning on every refactor.

### Specific suggestions
- Run-Tests.ps1: pass through pytest's last-line summary verbatim, or add a final aggregated line like `Summary: 819 passed, 0 failed, 5 skipped, 4 deselected, 5 xfailed`. (Doc-fixable improvement to the wrapper's print logic — not a memory item.)
- Lightweight Path L7: when the path mentions "if manual tests exist for the affected feature", add a one-liner pointing at the column or marker in test-tracking.md that distinguishes manual rows from automated.
