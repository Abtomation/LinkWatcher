## Improvement Suggestions

### What worked well
- **Batch-mode language in L1**: "copy the Item N section" gave explicit permission to expand scope mid-session without re-doing process from L1.
- **Test-only shortcut at L8**: One-line justification covering 7 evaluation criteria for all 6 items.
- **L5 + L11 checkpoints**: Caught the duplicate-investigation expansion before any irreversible bookkeeping was done. The mid-session pause to re-evaluate scope was the difference between resolving 1 TD and resolving 6.
- **Update-TechDebt.ps1 reliability**: 6/6 invocations succeeded with no parameter parsing issues, correct row movement, and full notes preservation.
- **Conditional L10 audit-loop closure**: The "only if the resolved TD's Source column or resolution notes reference a TE-TAR-* audit report" guard correctly skipped the Update-TestFileAuditState.ps1 step since the audit loop was already closed by PD-REF-196.

### What could be improved
- **Plan metadata not auto-updated when items are added**: When scope expanded from TD219 to 6 items, the frontmatter `debt_item`, `refactoring_scope`, and document `# Title` had to be hand-edited. The lightweight path's batch-mode instruction tells you to copy the Item section but doesn't mention updating these.
- **Update-TechDebt.ps1 verbose output for batch use**: 9 log lines × 6 invocations = ~54 lines of output. Could benefit from a `-Quiet` flag emitting only success/failure summary per invocation.

### Specific suggestions
- Consider adding to L1 in the lightweight path: "When adding an Item to an existing plan, also update the `debt_item` frontmatter field, `refactoring_scope` field, and document title to reflect the broader scope."
- Consider adding a `-Quiet` parameter to Update-TechDebt.ps1 for batch invocation scenarios.

## Additional Context

### Task-specific challenges
The principal challenge was bookkeeping archaeology: TD236-239 were created on 2026-04-22 from audit TE-TAR-066, then PD-REF-196 was created on 2026-04-28 with metadata `debt_item: TD215,TD216,TD217,TD218` — a *different* set of TD IDs covering the same audit findings. The work shipped under TD215-218; the open TD236-239 rows were left orphaned, with no linkage between the two sets. Without the user's "but also check c" prompt, this duplicate pattern would have remained undetected and TD236-239 would have continued accumulating. The lightweight path doesn't protect against this — it's a feature-request-level concern about how TD IDs get assigned when multiple agents/sessions log the same audit findings independently.

### Integration with other tools
- **Lightweight Path → New-RefactoringPlan.ps1**: L1's example invocation uses the exact parameter names the script accepts. No friction.
- **Lightweight Path → Update-TechDebt.ps1**: L10's example shows `-PlanLink` usage but TD220's resolved-row note effectively warns against it (PF-IMP-620). I deliberately omitted `-PlanLink` because of that warning — and the script worked correctly. This is a case where field-level tribal knowledge (in TD220's notes) is consumed correctly by an agent reading adjacent rows during investigation.
- **PD-REF plan archive structure**: Moving the completed plan to `doc/refactoring/plans/archive/` with `mv` worked cleanly; LinkWatcher is supposed to update any references, but this plan had no inbound links yet.
