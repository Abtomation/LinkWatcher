## Improvement Suggestions

### What worked well
- Mandatory impact analysis (Step 4) caught most surprises before destructive execution; the few discovered mid-flight were caught at a checkpoint and replanned cheaply
- Pilot checkpoint (Step 14) — testing the resolver in isolation before any moves was the right granularity
- File and Directory Move Procedure clearly defines the wait/grep/diagnose loop
- Forbidden-git-commands rule (CLAUDE.md) made the rollback-without-stash plan obvious
- Lock-file/PID + auto-detect resolver in the agnostic script worked first-try after the move

### What could be improved
- `New-StructureChangeState.ps1 -FromProposal -ChangeType Rename` is rejected; proposal-backed Rename would benefit from the same execution-tracking-only template that exists for other types
- `New-BugReport.ps1` doesn't expose `-Description`'s 500-char limit or `-DiscoveredBy`'s ValidateSet via `-?`; both surface only at runtime via the validator
- LinkWatcher's reference-update sweep can corrupt files when filename + path-segment overlap (PD-BUG-098); the move procedure should recommend a quick post-move grep for `<filename><path-tail>` corruption patterns, not just for orphaned old paths

### Specific suggestions
- Lift the `-FromProposal -ChangeType Rename` restriction OR document the rationale in the script's error message (not just "lightweight template already")
- Add `-Description (max 500 chars)` and `-DiscoveredBy (ValidateSet: …)` to New-BugReport's `-?` synopsis
- File and Directory Move Procedure: add a sub-step "after each move, grep for the corruption pattern `<old-path-tail><new-path-tail>` not just `<old-path>`"
- Pre-existing: validator's basename-only ignored-directory matching ([validator.py:18](src/linkwatcher/validator.py#L18) and [utils.py:84-86](src/linkwatcher/utils.py#L84-L86)) doesn't support multi-segment ignore values; SC-029 worked around this by using `linkWatcher` basename, but the limitation should be documented or lifted

## Additional Context

### Task-specific challenges
- **Mid-execution scope discoveries**: Three findings emerged after Step 4 impact analysis was complete: (1) the active `.linkwatcher-ignore` was at the project root, not in `LinkWatcher_run/`; (2) `install_global.py` regenerates the startup script from a Python f-string template; (3) a second `setup_project.py` copy lives in `deployment/`. Each required pausing for human direction before the plan could continue. Step 4's prescribed checks (reference grep, script audit, task audit) caught the path references but not the dynamic generation logic or the cross-coupling between `.linkwatcher-ignore` and `validation_ignore_file` config. A recommended addition to Step 4: "for each file being moved/deleted, search for code that *generates* it (f-strings, write_text, here-strings)."
- **LinkWatcher self-corruption**: Mid-move, LinkWatcher's reference-update sweep produced overlapping-substring rewrites in 17 files, including 2 load-bearing source files. PD-BUG-098 filed. Recovery was straightforward: stop LW, bulk-fix the consistent corruption pattern, surgically restore source files to bare filenames, continue with LW off and manual sweep.
- **Bulk-substitution overzealousness**: After fixing the corruption with LW off, my own bulk substitution in active doc files swept the state file and proposal too aggressively — replacing "Current Path" entries with "New Path" values, making historical context lossy. Should have excluded my own state/proposal docs from the bulk pass, since they describe before/after by design.

### Integration with other tools
- LinkWatcher (the tool being relocated) was simultaneously a *user* of paths AND a *modifier* of references describing those paths. This dual role is what triggered PD-BUG-098. PF-TSK-014's procedure correctly assumes LW is reliable for textual sweeps; the failure mode here was a previously unknown LW bug in overlapping-pattern handling.
- Validator's basename-only ignore matching (validator.py:18, utils.py:84-86) limited the scope of `validation_extra_ignored_dirs` defaults. SC-029 worked around this by using the leaf directory name `linkWatcher` instead of the full path `process-framework-local/tools/linkWatcher`. This pre-existed SC-029 but became visible because the user originally chose the more precise multi-segment value (Option A). Worth documenting as a known limitation or fixing in a follow-up.
