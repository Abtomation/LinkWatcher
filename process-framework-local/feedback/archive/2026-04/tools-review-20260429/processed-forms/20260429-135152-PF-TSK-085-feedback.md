## Improvement Suggestions

### What worked well
- Lifecycle status semantics and audit-gate description in PF-TSK-085 are unambiguous.
- The four invocation patterns in step 10 of PF-TSK-085 cover every realistic transition.
- Update-PerformanceTracking.ps1 logging makes the wrong summary-counter bug instantly visible (the script's own log line was what surfaced it).
- `performance_db.py regressions` correctly ignored cosmetic deltas (rework-induced unit changes, throughput sign convention) and reported zero regressions.
- The new feature (mean-of-N-runs notes captured in DB) gives future audits clear provenance.

### What could be improved
- **Update-PerformanceTracking.ps1 Summary recalculation matched ✅ across the whole row line**, double-counting every row whose Audit Status is `✅ Audit Approved`. Fixed in this session by parsing the Status cell (column 4) directly. The same line-wide emoji-match pattern appeared only in this one location (verified via grep), so the fix is local.
- **PF-TSK-085 lacks measurement-strategy guidance**: should the operator do 1 run, N runs averaged, N runs with median, or N runs dropping the cold-cache run? I had to ask the user (who said "average") and then propose three options at the checkpoint.
- **PF-TSK-085 ambiguous on tolerance breaches with stale baselines**: when a row is `⚠️ Needs Re-baseline` AND the latest LastResult exceeds the tolerance column, the task says "tolerance breach → Bug Triage" but in this case the breach comes from an outdated baseline expectation, not a real regression. Need a decision rule: was this measured against a legitimate prior baseline, or a since-invalidated one?
- **`performance_db.py trend` cosmetic issues** (display only, regression detection is correct):
  - Labels all `+%` deltas as "degradation" regardless of unit — wrong for throughput metrics where higher = better (BM-001, BM-004).
  - Tolerance values printed in trend output don't match the tracking file's Tolerance column (BM-002 `<5.0s` vs `<3s`; BM-007 `<2.0s` vs `<1.8s`; BM-008 `<2.0s` vs `<0.02s`) — looks like a stale embedded config.
  - When a test definition changes (e.g., BM-002 went from 1000 to 10000 ops between baselines), trend shows misleading raw deltas (`+1533%`). Could optionally tag entries with a "test-revision" key and reset trend deltas at revision boundaries.

### Specific suggestions
- **Add measurement-strategy guidance to PF-TSK-085**: a short section under Execution describing default (3-run mean), when to use median (high variance), when to drop cold-cache run 1 (warm-cache steady-state baseline), and when to record per-run vs. mean in the DB.
- **Add a tolerance-breach decision tree to PF-TSK-085 step 9**: distinguish "current run breaches the tolerance" (real regression) from "prior LastResult breaches tolerance against a stale baseline" (informational only — re-baselining clears it).
- **Update-PerformanceTracking.ps1 fix shipped this session** — preserved the existing reorder-vs-column-parse trade-off in favor of column-parse for robustness against future audit-column emoji additions. Comment in the code explains why.
- **Fix `performance_db.py trend` display** to use sign convention from unit (files/sec → "+ = improvement"; seconds/ms → "+ = degradation"), and source the tolerance from `performance-test-tracking.md` rather than any embedded copy.
