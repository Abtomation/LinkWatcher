## Improvement Suggestions

### What worked well
- L1's effort gate kept the recommendation defensible — independently verifying the TD description by reading the actual code (instead of accepting the TD note at face value) led to the concrete fix-by-analogy approach (port Phase 2's PD-BUG-094 pattern verbatim).
- L4's coverage assessment correctly identified that existing integration tests prove the *current* behavior with the collection-layer filter, but not the idempotency property of `_replace_at_position` in isolation. This led to 3 unit tests targeting exactly the property the guard is protecting.
- L8's grep-based justification for design docs (TDD/FDD/ADR/Test spec) was efficient — methods referenced only at dispatch level genuinely don't need updates for internal-implementation refactors.
- The two-checkpoint discipline (L5 plan, L11 results) is well-placed.

### What could be improved
- Integration narratives are mentioned in the L8 list but warrant slightly stronger guidance: they are the doc type most likely to quote private-method internals verbatim, so they go stale on internal-implementation refactors that other doc types don't notice. A targeted callout would catch more drift.
- Validate-TestTracking.ps1 output mixes pre-existing and new warnings without visual separation. A "delta-from-last-run" mode would surface only the warnings the current session may have introduced.

### Specific suggestions
- **Lightweight Path L8**: Add a sentence near the Integration Narrative bullet: *"Grep for the changed method's exact source strings (operators, keywords, regex literals), not just the method name — narratives often quote implementation lines verbatim and these are the doc fragments most likely to go stale on internal refactors."*
- **Validate-TestTracking.ps1**: Optional `-DeltaFromLastRun` switch that diffs current warnings against a stored snapshot from the last successful run, so the operator sees only what they may have caused.
