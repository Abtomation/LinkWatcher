## Improvement Suggestions

### What worked well
- **Self-bootstrapping correctness**: Phase 2's `Assert-LineInFile` was reused by Phase 3's `_Write-SoakStateRaw` for read-after-write verification. The new infrastructure validated itself before any pilot script touched it.
- **Snapshot/restore smoke test pattern**: The Phase 3 smoke test snapshotted `script-soak-tracking.md` before testing and restored it after, with an explicit T18 verifying bit-for-bit equality. This left zero test data in the production state file. Worth replicating as a pattern for any future smoke tests that touch persistent state.
- **Iterative checkpoint summaries**: Each phase's end-of-phase summary surfaced concerns the human partner could act on. The ID-prefix mismatch fix happened *because* the Phase 3 summary mentioned the mismatch as a known issue; without that visibility, it would have stayed deferred.
- **Two-registry split (main vs. local)**: Adding `PF-SST` to the main registry was a 5-minute fix exactly because the registry architecture already supported the distinction.

### What could be improved
- **Concept template should pre-flight ID prefix locality**: The Framework Extension Concept template (PF-PRO-028 used here) does not prompt the author to evaluate whether new state file locations are consistent with their assigned ID prefix's registered directory. This was the proximate cause of the mid-session migration. Adding a 1-line pre-flight question would have caught it during Phase 1.
- **PF-STA description is misleading**: `process-framework-local/PF-id-registry-local.json` describes PF-STA as "(framework only)" but every directory mapping is `process-framework-local/...`. Either the description is wrong or the directories are wrong; either way a fresh reader will be confused. Quick fix: change description to "Project-local State Tracking".
- **Multi-phase calendar session**: This session ran Phase 2 + Phase 3 + an unplanned ID migration. The state file's Session Tracking section still labels them as "Session 2" and "Session 3" but they happened on the same calendar day. The labeling implicitly endorses one-phase-per-calendar-session; the actual pattern should be documented explicitly.
- **Phase 4 testing strategy not pre-planned**: Phase 4's "Run for real" test step creates persistent PD-INT and PD-UGD artifacts. The concept does not say whether these are kept (they're "real" docs), reverted (LinkWatcher-monitored — risky), or run against a scratch project. Decision should be made in Phase 1 concept, not on the morning of Phase 4.

### Specific suggestions
- Add a "State File Locality Pre-Flight" sub-section to the framework-extension concept template:
  ```
  Are any new state files introduced by this extension?
  If yes, for each:
    - Will the file be project-local or shareable across projects?
    - Does the assigned ID prefix's directory mapping match that locality?
    - If not, the prefix needs to be added to the appropriate registry.
  ```
- Update PF-STA's description in `process-framework-local/PF-id-registry-local.json` from "Process Framework - State Tracking (framework only)" to "Process Framework - Project-Local State Tracking" so the description aligns with the directories.
- Add a "Side-Effect Strategy" sub-section to framework-extension concepts that involve pilot adoption: explicit decision about whether real-mode tests use scratch projects, are run with `-WhatIf` only, or produce persistent artifacts that are accepted as legitimate.

## Additional Context

### Task-specific challenges
This session combined Phase 2 (helpers), Phase 3 (soak module + new state file + new shareable directory), and an unplanned mid-session ID-prefix migration. The challenges were:
- **Self-bootstrapping risk**: Phase 3's soak module uses Phase 2's `Assert-LineInFile` for its own writes. If Phase 2 had a latent bug, Phase 3 would fail in unexpected ways. The 16-case Phase 2 smoke test before starting Phase 3 was load-bearing for confidence.
- **Persistent state-file safety**: The Phase 3 smoke test had to exercise Register/Test/Confirm against the production `script-soak-tracking.md`. The snapshot/restore harness (T18 verifies bit-for-bit equality) was the right answer but required deliberate design.
- **Line-ending preservation**: PowerShell's default Set-Content behaviors around CRLF vs LF on Windows could have introduced subtle text-diff churn in the state file. Used `[System.IO.File]::ReadAllText` / `WriteAllText` with explicit UTF-8-no-BOM and a `_Get-LineEnding` helper to round-trip without mutation.
- **WhatIf bypass coverage**: had to test all three write paths (Register, Test-with-hash-reset, Confirm) under WhatIf to confirm no state writes — got coverage in T5/T12/T13.

### Integration with other tools
- **`Common-ScriptHelpers.psm1` facade**: adding the new sub-module was 2 edits (one line in `$SubModules`, one new section in static fallback). The facade pattern absorbed the addition cleanly with no breaking changes for existing callers.
- **`Get-ProjectRoot` from Core.psm1**: ExecutionVerification.psm1 calls `Get-ProjectRoot` to locate the state file. This works because the facade loads Core before ExecutionVerification, exporting `Get-ProjectRoot` to the session. Cross-sub-module dependency was not explicit but worked correctly.
- **`Get-FileHash -Algorithm SHA256`**: built-in PowerShell cmdlet, zero friction. Dependency-free hash computation for the soak counter's auto-reset detection.
- **PF-documentation-map.md**: doc-map entries for the new sub-module and state file were added manually; no automation script handles this for arbitrary new artifacts. Two edits, ~30 seconds. Not a friction point for one-off additions but would be nice to have a generic `Add-DocMapEntry` helper for the broader rollout (Phase 5+).
