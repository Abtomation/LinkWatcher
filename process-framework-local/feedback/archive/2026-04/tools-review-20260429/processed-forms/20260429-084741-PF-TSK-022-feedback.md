## Improvement Suggestions

### What worked well
- Lightweight Path L2's DA-category guidance ("trace the root cause of the drift before fixing it") forced upfront analysis that surfaced the TD128/TD129 + PD-BUG-050/091 origin chain and the sibling refactor precedent (PD-REF-195/198). Without that step I would have applied the literal pseudocode fix without noticing PD-REF-198's prior phase-granularity decision.
- Lightweight Path L3/L7 documentation-only exemption let me skip test baseline + regression cleanly. The exemption is well-named and explicit, no judgment call needed.
- The path's L8 grep-each-doc-type checklist surfaced the line 507 prose drift even though TD226's literal scope was just the pseudocode block.
- Update-TechDebt.ps1's three-mode design (Add / Status-only / Completion) keeps semantics crisp — the column-mapping comment in the help text was particularly useful.

### What could be improved
- Lightweight Path L8 documentation-only shortcut wording is slightly broken when the doc-only change IS itself in a design document (TDD/FDD/ADR). The shortcut text reads: *"design and state documents do not need updates for [description of change]"* — which is logically inverted when the change is to a TDD. I had to mentally invert: item 2 (TDD) = ✅ this IS the change, items 1, 3-7 = N/A by shortcut.
- New-RefactoringPlan.ps1's mutually-exclusive flag error doesn't disambiguate. Adding two sentences to the error ("Use `-DocumentationOnly` for doc-only refactors. The Lightweight/Performance/DocumentationOnly flags select template variants and combine implicitly with the Lightweight Path workflow.") would have saved one round-trip.
- Pattern signal: Section 4.3 of PD-TDD-023 has now been refactored 3 times in 3 weeks (TD220, TD228, TD226). Its "reverse-engineered from source code analysis" provenance (line 17 of the TDD) makes it drift-prone whenever handler.py is touched. Captured in PD-REF-204's Remaining Tech Debt as "if a fourth surfaces, file a process improvement". Not raising as PF-IMP per this form's explicit guidance to leave that to PF-TSK-009/010.

### Specific suggestions
- **[Lightweight Path L8 shortcut wording]** — Add a clarifying note: *"When the documentation-only change is itself a design-document edit (TDD/FDD/ADR), mark item 2/4/5 (whichever applies) as the change itself and apply the shortcut to the remaining items."*
- **[New-RefactoringPlan.ps1 error message]** — Replace `"-Lightweight, -DocumentationOnly, and -Performance are mutually exclusive. Use at most one."` with a longer message that explains each flag's purpose. Even a one-line trailing hint ("for doc-only refactorings, use `-DocumentationOnly` alone") would resolve the ambiguity.
- **[Documentation-only refactoring plan template]** — Optionally add a "Design Decisions Made During Plan Drafting" section between Refactoring Goals and Current State Analysis, for cases where the doc fix has multiple plausible rendering options. I added one ad-hoc here (Option A/B/C for phase granularity) and it was useful for the L5 checkpoint.
