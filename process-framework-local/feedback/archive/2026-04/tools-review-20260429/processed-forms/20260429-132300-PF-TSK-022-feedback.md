## Improvement Suggestions

### What worked well
- Lightweight Path's mid-process checkpoint structure caught the unit-mismatch regression at L7 and prompted the user discussion that produced Option B (bundling TD249's PH-008 sub-item).
- Audit-flagged-TD-closure callout in L10 explicitly handled the partial-closure case for TE-TAR-070 — no risk of erroneously marking Audit Approved.
- Test-only shortcut in L8 collapsed an otherwise lengthy doc/state checklist to a single justified N/A.
- TD249's existing notes (which named its blockers and explicitly authorized "or remove entirely" for the PH-008 peak threshold) made Option B's scope expansion safe and well-defined — the prior author had foreseen this exact case.

### What could be improved
- **Update-TechDebt.ps1 doesn't support description-only edits.** When a refactoring resolves a sub-item of another open TD (without resolving the whole TD), the amendment must be done by manual file edit. Adding a `-EditDescription` / `-EditNotes` mode would close this gap.
- **New-RefactoringPlan.ps1 doesn't substitute `-DebtItemId` into the Item 1 heading.** The script knows the value (it's in frontmatter and the post-creation message) but the template heading remains `[Debt Item ID] — [Brief Description]`. A small string substitution would save one manual Edit.

### Specific suggestions
- Add `-EditDescription` and `-EditNotes` modes to `Update-TechDebt.ps1` for in-place amendments to open TD items, with the same validation/atomicity guarantees as the resolve path. Use case: resolving a sub-item of another TD without resolving the whole thing.
- In `New-RefactoringPlan.ps1`, when `-DebtItemId` is provided, substitute it into the `Item 1: [Debt Item ID] — [Brief Description]` heading. Also consider auto-stamping the Test Baseline / Test Result placeholders with `[Fill after L3 baseline]` / `[Fill after L7 regression]` for clarity.

## Additional Context

### Task-specific challenges
The literal TD247 prescription ("switch to `process.cpu_percent`") produced a unit mismatch on multi-core hosts because `process.cpu_percent` returns `cores × 100%` while the existing thresholds (`assert max_cpu < 95`, `assert avg_cpu < 80`) were calibrated for `[0, 100]`-scale `psutil.cpu_percent` values. The L7 regression run revealed `Peak: 117.6%, Average: 64.2%` — a real failure caused by my change. This wasn't a doc gap or tool gap — it was an execution failure during L1 verification. The Step 1 callout to "independently verify TD descriptions ... is this the COMPLETE picture?" applies precisely to this case. I should have run a units check on the substitution. The process recovered cleanly via L7's regression-vs-baseline diff and the user-facing checkpoint that followed.

No actual conflict — just had to re-read the file to find current line numbers.
