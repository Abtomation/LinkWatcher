## Improvement Suggestions

### What worked well
- Validate-StateTracking.ps1 surface pattern is consistent and easy to extend; new Surface 15 slotted in cleanly with ~140 LOC and no refactoring of existing surfaces.
- Step 2 evaluation table caught the dual-vocabulary problem before any code was written; without it I would have implemented a naïve string-equality check that flagged every active feature row.
- Update-ProcessImprovement.ps1 handles InProgress→Completed lifecycle atomically; no manual table editing risk.
- Tool change logging via feedback_db.py creates a durable audit trail tying the diff to the IMP for trend analysis.

### What could be improved
- New-ProcessImprovement.ps1 500-character `-Description` cap is enforced as a runtime error after invocation; would be friendlier to either document it in `-?` synopsis or auto-truncate-with-warning.
- feedback_db.py `--tool` parameter accepts arbitrary strings without validating against a known-tools list — easy to silently log a wrong/typo'd tool ID. A `query --list-tools` subcommand or schema validation against the docs map would prevent this.
- feedback_db.py success message could surface the resolved DB path; central-DB redirection via domain-config.json is invisible by default and led to ~5min of wasted "where did my entry go?" diagnosis.
- Validate-StateTracking.ps1 Surface 6 (FeatureDeps) crashes when run from this project (stale path inside Update-FeatureDependencies.ps1 looking in `.git/objects/...`); this halts the full validation chain and prevents `-Surface All` from completing. Pre-existing — not caused by PF-IMP-573 — but worth its own bug entry.

### Specific suggestions
- **PF-IMP suggestion**: Add `--list-tools` subcommand to feedback_db.py that prints distinct tool_doc_ids from `tool_changes` + `tool_ratings`, enabling agents to verify the canonical ID before logging.
- **PF-IMP suggestion**: Have feedback_db.py print the resolved DB path on every command (or at least on `log-change` success).
- **PF-IMP suggestion**: Have New-ProcessImprovement.ps1 print the description char-count on validation failure so the agent knows by how much to compress (current error says only "762 > 500").
- **Bug suggestion**: Investigate Update-FeatureDependencies.ps1 path resolution — it's looking in `.git/objects/<hash>/doc/state-tracking/features` instead of the working tree.

## Additional Context

### Task-specific challenges
The IMP description ("aggregate test-tracking statuses and check against feature-tracking") implicitly assumed the two files used the same status vocabulary. Reading the actual data exposed they don't — the writer (Update-TestFileAuditState.ps1) and the legend (feature-tracking.md) define different text per emoji, and the in-table values had been edited back to the legend vocabulary since the writer last ran. This forced an architectural decision (status normalization vs. vocabulary unification vs. hybrid) that wasn't anticipated by the IMP. The Step 5 checkpoint exists for exactly this reason — surfacing it to the human before implementation rather than during.

### Integration with other tools
