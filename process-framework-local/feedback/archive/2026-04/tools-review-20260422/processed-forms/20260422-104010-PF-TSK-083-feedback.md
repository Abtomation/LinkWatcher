## Improvement Suggestions

### What worked well
- Two-checkpoint task structure (Preparation + Review) caught the scope question (include 0.1.3 or not?) early
- The customization guide's "Include only features that actively participate — supporting features belong only if they have workflow-specific behavior" criterion is a sharp decision tool
- Inline HTML comments in the template serve as AI-readable instructions that disappear from rendered output
- "Not applicable" replacement text for conditional sections (Callback/Event Chains, Configuration Propagation) removes ambiguity
- Reading a sibling narrative (PD-INT-003 dry-run-mode) as a concrete example was faster than following the guide abstractly

### What could be improved
- TD221 and TD222 track the same defect (PD-documentation-map.md heading regex) — they should be merged into one tech debt item to avoid double-counting in tech-debt dashboards
- The script's warning message ("Could not find section '### Integration Narratives'") could tell the user exactly what the expected heading is vs what was found, so the manual fix is one copy-paste instead of a grep
- The Data Flow Sequence section could benefit from a reminder in the template that line numbers should be captured "as of YYYY-MM-DD" to signal their staleness risk (the template already hints at this for callback registration locations but not for data-flow step references)

### Specific suggestions
1. **Merge TD221 + TD222** into one entry. Same root cause (regex mismatch with actual heading), same file (`New-IntegrationNarrative.ps1`), same fix scope.
2. **Fix the doc-map heading regex** in `New-IntegrationNarrative.ps1` to match the actual `../../../../feedback-forms### \`technical/integration/\` — Integration Narratives` heading. Low effort (0.5h-S), high frequency (runs every narrative).
3. **Enhance the script warning** to show the expected-vs-found heading, making the manual fix trivial while the bug remains open.
