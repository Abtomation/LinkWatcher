## Improvement Suggestions

### What worked well
- The "verify against source code, do NOT trust TDDs alone" rule in PF-TSK-083 is critically important and catches real divergences (TD226, TD228 here).
- Checkpoint placement (scope → draft → finalization) is appropriate.
- Using an existing narrative (PD-INT-002) from the same feature set as a structural reference greatly accelerated scoping.
- `-ListDims` on `Update-TechDebt.ps1` is a nice UX touch for discovering valid values.

### What could be improved
- **Task doc example command for `Update-TechDebt.ps1` is stale** — uses `-Title` / `-Severity` parameters that don't exist. Actual params: `-Description`, `-Dims`, `-Priority`, `-EstimatedEffort`. Task step 7 line 100 in `process-framework/tasks/02-design/integration-narrative-creation.md` should be corrected.
- **New-IntegrationNarrative.ps1 hardcoded section header** (TD230). Should match the actual PD-documentation-map header `../../../../feedback-forms### technical/integration/ — Integration Narratives`.

### Specific suggestions
- Fix the Update-TechDebt.ps1 example in the task doc (file: `process-framework/tasks/02-design/integration-narrative-creation.md`, step 7 — swap `-Title "..." -Category "..." -Severity "..."` for `-Description "..." -Dims "..." -Location "..." -Priority "..." -EstimatedEffort "..."`).
- Consider having `New-IntegrationNarrative.ps1` read the section header via a more tolerant match (regex on `^### .*Integration Narratives`) to avoid the silent-warn path.
