## Improvement Suggestions

### What worked well
- Moving directories one at a time and verifying LinkWatcher results after each was the right approach
- LinkWatcher handled the vast majority of reference updates automatically (markdown links, backtick paths, bare paths, PS1 file references)
- The PF-id-registry split cleanly separates blueprint schema from project-local counters
- The `process-framework-local/` naming makes the boundary obvious

### What could be improved
- Initial assumption that LinkWatcher couldn't handle code blocks was wrong — wasted time planning manual updates that weren't needed
- Read the wrong log file after a restart, leading to false conclusion that LinkWatcher was dead
- The impact analysis (Phase 3) was overly detailed given that LinkWatcher handles most updates

### Specific suggestions
- For future structure changes involving directory moves: try one small directory first to calibrate how much LinkWatcher handles, then scale up
- PD-BUG-075 should be fixed — phantom link targets in the database inflating directory move detection counts

## Additional Context

### Task-specific challenges
- PD-BUG-075 (phantom link targets inflating directory move detection) caused initial feedback/ and evaluation-reports/ moves to fail, requiring investigation and restart
- Log rotation caused reading stale log files, leading to incorrect conclusions about LinkWatcher's state
- Large number of files in feedback/ (70+) plus phantom entries (1400+) stressed the move detection system

### Integration with other tools
- LinkWatcher was the primary workhorse for reference updates — the structure change task should document this as a first-class migration strategy
- IdRegistry.psm1 needed updating to support the split registry — this was a clean change thanks to the existing module architecture
