## Improvement Suggestions

### What worked well
- Pre-concept analysis sub-steps (a-e) prevented premature design decisions
- Human checkpoints caught a real design issue (coverage dedup would create regression blind spot)
- Extension type classification (Creation/Modification/Hybrid) helped select appropriate template sections
- Concept document structure organized a complex 27-artifact extension into a reviewable format

### What could be improved
- New-StructureChangeState.ps1 template is designed for file migration, not framework extensions — modification-heavy extensions need a different template variant
- New-FrameworkExtensionConcept.ps1 template includes sections not applicable to one-time extensions
- No built-in way to bundle related IMP items — had to manually decide and document the bundling rationale

### Specific suggestions
- Add a `-Variant FrameworkExtension` to New-StructureChangeState.ps1 (analogous to New-TempTaskState.ps1's -Variant) that generates a template with Component Inventory, Session Plan, and Session Log sections instead of Pilot/Rollback/Metrics
- Add a `-ExtensionType Modification` parameter to New-FrameworkExtensionConcept.ps1 to generate a leaner template without New Tasks Required, New Permanent State Files, and Example Use Cases sections
