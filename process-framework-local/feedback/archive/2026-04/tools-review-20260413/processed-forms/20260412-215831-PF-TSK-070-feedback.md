## Improvement Suggestions

### What worked well
- Scripted test pipeline (Run-E2EAcceptanceTest.ps1) handles the full lifecycle for exit-code-0 tests
- New-BugReport.ps1 is efficient and well-integrated
- Master test file structure made the validation sequence clear
- Test case fixtures were well-designed to isolate specific behaviors

### What could be improved
- Run-E2EAcceptanceTest.ps1 needs support for expected non-zero exit codes (PD-BUG-089)
- Update-TestExecutionStatus.ps1 needs per-case filtering when -TestCase is specified

### Specific suggestions
- Add `expected_exit_code` field to test case YAML frontmatter (default: 0); orchestrator reads it and compares
- When -TestCase is passed to Update-TestExecutionStatus.ps1, only update that specific row, not the entire group

## Additional Context

### Task-specific challenges
Validation-mode E2E tests are unusual because the expected exit code is non-zero. The existing orchestrator assumes exit code 0 = success, which doesn't hold for this test group. This required manual workarounds: re-running validation independently, checking reports manually, and editing tracking files by hand.

### Integration with other tools
New-BugReport.ps1 integrated seamlessly — bugs filed and tracked. The orchestrator's integration with Update-TestExecutionStatus.ps1 worked well for passed tests but couldn't handle the error-path cases.
