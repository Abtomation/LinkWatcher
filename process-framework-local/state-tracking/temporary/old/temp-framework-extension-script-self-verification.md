## Notes and Decisions

### Key Decisions Made (during Phase 1)

- **Single regex `-Pattern` parameter for `Assert-LineInFile`** (not specialized helpers like `Assert-RowInTable`). Rationale: YAGNI; single regex with `-MinOccurrences` covers all known cases.
- **`Confirm-SoakInvocation` as function call, not Read-Host**. Rationale: agent-driven verification works in non-interactive contexts; Read-Host blocks pipelines.
- **Counter hardcoded at 5** (no `-InitialCounter` parameter). Rationale: arbitrary anyway; configurable via concept revision later if needed.
- **Hash-based auto-reset on script body change**. Rationale: agents/humans should not have to remember to call reset manually; SHA256 detection is a one-line PowerShell built-in.
- **WhatIf bypass**: `Test-ScriptInSoak` returns `$false`, `Confirm-SoakInvocation` no-op. Rationale: WhatIf runs don't produce real edits; counting them would distort the soak.
- **Forward-only adoption**: existing battle-tested scripts not retroactively soaked. Rationale: avoids prompt fatigue and undermines the "new/reworked" signal.
- **State file location**: `process-framework/state-tracking/permanent/` (NEW shareable directory) instead of `process-framework-local/`. Rationale: scripts are shared across projects; soak state should be too.
- **Script ID = relative path from project root** (Option α). Rationale: stable, unique, no new ID-registry infrastructure needed. Option β (`PF-SCR` prefix) deferred to PF-IMP-601.
- **Two registration hooks** (PF-TSK-026 + PF-TSK-001 Session 2). Rationale: both tasks can create new scripts; single hook would leave PF-TSK-001-created scripts unregistered.
- **Two-script pilot** (`New-IntegrationNarrative.ps1` + `New-Handbook.ps1`). Rationale: both share the doc-map silent-success failure pattern; armoring both validates the design across more than a single point.
- **Broader rollout = one master IMP after Session 3 pilot proves value**. Rationale: don't pre-commit to per-script IMPs; one triage pass after value is demonstrated.
- **No TTL/expiry on stale soaks**. Rationale: `Get-SoakStatus` exposes stale entries during periodic Tools Review sessions; TTL adds complexity without proportional value.
- **PS-only scope**. Rationale: Python scripts (e.g., `feedback_db.py`) are out of scope; would need separate Python helper module.
