## 1. Feature Overview

### Feature Description

The Python Parser extracts file references from `.py` files by scanning for three kinds of references: (1) quoted string literals that look like file paths, (2) file path references embedded in comments, (3) local `import`/`from` statements converted from Python dot notation to slash paths. A hard-coded stdlib skip list filters out common standard library module names before attempting local import detection.

For import detection, `_looks_like_local_import()` applies a heuristic: imports matching known project directory prefixes (e.g. `src`, `lib`) or containing 3+ segments are considered local. Dot notation is converted to slash paths (e.g. `from src.utils.helpers import X` â†’ `src/utils/helpers`) for storage in `LinkReference.link_target`.

### Business Value

- **User Need**: Python source files contain import statements and string paths referencing other files; these must be updated when source files move
- **Business Goal**: Track local Python import paths so that module moves can be reflected in all referencing files
- **Success Metrics**: Local imports detected correctly; stdlib imports not falsely tracked; quoted string paths in Python code found

### Scope

**In Scope**:

- Quoted string literals that pass `_looks_like_file_path()`
- File paths in comment lines
- Local `import`/`from` statements (dot notation â†’ slash path conversion)
- Stdlib skip list for common module names

**Out of Scope**:

- Dynamic imports (`importlib.import_module()`, `__import__()`)
- `sys.path` manipulation detection
- Third-party package import tracking

## 7. Design Decisions

### Decision 1: Triple-Strategy Detection (Quoted Strings + Comments + Imports)

**Date**: 2026-02-18 (discovered)
**Context**: Python files contain file references in three distinct contexts, each requiring a different detection approach: string literals that are file paths, file paths embedded in comments, and import statements whose module paths map to source files.

**Options Considered**:

1. Regex-only scanning for quoted paths: Misses import statements entirely
2. AST-based parsing: Accurate but heavyweight; requires import infrastructure; overkill for path extraction
3. Three independent regex passes per file (chosen): Each mode handles one reference context; independent maintenance

**Decision Made**: Three regex passes â€” quoted string literals, comment lines, import/from statements (Option 3)

**Rationale**: Each detection mode requires different patterns. Independent passes are readable and testable separately. No external dependency needed.

**Implications**:

- Dynamic imports (`importlib.import_module()`) are out of scope â€” accepted limitation
- Order of passes doesn't affect results; all three run regardless of what earlier passes found

---

### Decision 2: Stdlib Skip List with Local Import Heuristic

**Date**: 2026-02-18 (discovered)
**Context**: Python imports are either stdlib/third-party (not tracked) or local (tracked). Without filtering, every `import` statement would generate a false reference.

**Options Considered**:

1. Track all imports: Too many false positives; most are stdlib or third-party
2. Hard-coded stdlib skip list + heuristic (chosen): Skip list for common stdlib names; heuristic (known prefix OR 3+ segments = local) for ambiguous cases
3. Require project configuration listing local prefixes: Adds setup burden

**Decision Made**: Stdlib skip list + `_looks_like_local_import()` heuristic (Option 2)

**Rationale**: No project configuration required. Practical accuracy for typical projects. False-negative rate on unusual import structures is an acceptable trade-off.

---

### Decision 3: Dot-to-Slash Module Path Conversion

**Date**: 2026-02-18 (discovered)
**Context**: Python module notation (`src.utils.helpers`) is not a file system path. It must be converted before storage in `LinkReference.link_target` so path matching in the updater can function.

**Options Considered**:

1. Store raw module notation: Incompatible with file-system path matching in updater
2. Convert dot to slash (chosen): Simple; produces `src/utils/helpers` for matching against actual file paths
3. Resolve to absolute path at parse time: Would require project root knowledge; too coupled

**Decision Made**: Replace `.` with `/` in import module path (Option 2)

**Rationale**: Simple, reversible transformation. Enables consistent path matching without coupling the parser to project structure knowledge.

### Implementation Patterns Used

**Structural Pattern**:

- Pattern: Template Method (via `BaseParser`)
- Why: Common file-read infrastructure in base; Python-specific detection in `PythonParser`
- Where: `BaseParser._safe_read_file()`; three detection passes in `PythonParser`

**Classification Pattern**:

- Pattern: Strategy (three detection modes run independently)
- Why: Each mode targets a different Python reference context; independent pass structure is maintainable
- Where: `_extract_string_refs()`, `_extract_comment_refs()`, `_extract_import_refs()`
