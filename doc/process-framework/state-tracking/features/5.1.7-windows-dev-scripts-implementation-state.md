## 1. Feature Overview

### Feature Description

Provides Windows-native development command scripts via `dev.bat` (Windows batch) and `Makefile` (cross-platform make). Both expose identical command sets: `install`, `install-dev`, `test`, `test-quick`, `test-all`, `coverage`, `lint`, `format`, `type-check`, `clean`, `build`, `pre-commit`, `ci-test`, and `dev-setup`. The `dev.bat` uses goto-label routing (`if "%1"=="test" goto test`) for Windows CMD compatibility.

The Makefile adds additional targets: `help` (default, lists all commands), `docs` (placeholder), `release-check` (clean â†’ lint â†’ type-check â†’ test-all â†’ build â†’ twine check). Both scripts delegate to `run_tests.py` for test execution and to standard Python tools (black, isort, flake8, mypy) for quality checks. Manual test scripts (`manual_test/scripts/deploy.ps1`, `manual_test/scripts/setup.ps1`) and placeholder files provide example deployment/config script structures.

### Business Value

- **User Need**: Single-command access to common development tasks on Windows
- **Business Goal**: Lower the barrier to running tests, formatting, and building locally
- **Success Metrics**: `dev test`, `dev lint`, `dev build` all work from project root

### Scope

**In Scope**:

- `dev.bat` â€” Windows batch command router (20 commands)
- `Makefile` â€” GNU Make targets (16 phony targets)
- `manual_test/scripts/deploy.ps1`, `manual_test/scripts/setup.ps1` â€” manual test deployment scripts
- `manual_test/scripts/automation/placeholder.txt`, `manual_test/scripts/config/placeholder.txt` â€” placeholder files

**Out of Scope**:

- `run_tests.py` test runner logic (covered by 4.1.1)
- Pre-commit installation (covered by 5.1.5)
- CI pipeline (covered by 5.1.1)

## 7. Design Decisions

### Decision 1: Dual Script Approach (Makefile + dev.bat)

**Context**: Windows developers may not have GNU Make installed; need a native Windows alternative

**Decision Made**: Provide both `Makefile` (for developers with Make) and `dev.bat` (native Windows CMD) with identical command sets

**Rationale**: `Makefile` is standard for Unix-familiar developers and provides dependency-based execution. `dev.bat` provides the same commands without requiring Make installation. Both delegate to the same underlying tools.

**Implications**:

- Command sets must be kept in sync between both files
- `dev.bat` uses goto-label routing instead of Make's target system
- Makefile provides additional features (phony targets, `release-check` dependency chain)

### Decision 2: Compound dev-setup Target

**Context**: New developers need to set up their environment with multiple steps

**Decision Made**: Provide a `dev-setup` target that chains `install-dev` + `pre-commit` into a single command

**Rationale**: Reduces new developer onboarding to a single command. Follows the principle of least surprise â€” after running `dev dev-setup`, the environment is fully ready.

**Implications**:

- Single entry point for new developer setup
- Makefile uses `dev-setup: install-dev pre-commit` dependency chain
- dev.bat uses `call :install-dev` + `call :pre-commit` sequential calls
