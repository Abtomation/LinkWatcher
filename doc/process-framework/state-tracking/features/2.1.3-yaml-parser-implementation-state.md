## 1. Feature Overview

### Feature Description

The YAML Parser implements file reference extraction for `.yaml` and `.yml` files. Rather than scanning raw text with regex, it first loads the full document using `yaml.safe_load()` to produce a Python data structure (dicts, lists, scalars), then recursively traverses that structure looking for string values that look like file paths via `_looks_like_file_path()`. This approach handles all YAML formatting styles (flow, block, multi-line) transparently.

When a matching string value is found, the parser uses `_find_next_occurrence()` to locate its position in the raw file content, advancing past previously found positions to correctly handle duplicate values. If `yaml.safe_load()` raises `yaml.YAMLError`, the parser falls back to `GenericParser` for line-by-line text scanning.

### Business Value

- **User Need**: YAML configuration files frequently contain file paths as values; these must be tracked and updated when referenced files move
- **Business Goal**: Accurately extract file references from any valid YAML structure, regardless of nesting depth or formatting style
- **Success Metrics**: File path values found at all nesting levels; duplicate values correctly tracked; broken YAML files still parsed via fallback

### Scope

**In Scope**:

- Recursive traversal of dicts, lists, and scalar strings
- `_looks_like_file_path()` validation for string values
- `_find_next_occurrence()` for correct position tracking of duplicate values
- `GenericParser` fallback for invalid/unparseable YAML

**Out of Scope**:

- YAML anchor/alias analysis
- Key-based filtering (all string values are candidates regardless of key name)
- YAML schema validation

## 7. Design Decisions

### Decision 1: Structural Parsing (yaml.safe_load) Over Regex Scanning

**Date**: 2026-02-18 (discovered)
**Context**: YAML supports many equivalent formatting styles (flow, block, multi-line scalars). Regex scanning of raw YAML text would miss values in non-canonical forms and generate false positives from YAML syntax tokens.

**Options Considered**:

1. Regex scan raw YAML text: Fragile; misses multi-line scalars; YAML syntax tokens generate false positives
2. `yaml.safe_load()` + recursive traversal (chosen): Format-agnostic; all string values reachable regardless of style
3. YAML event-based streaming (`yaml.parse()`): More complex; no benefit over safe_load for this use case

**Decision Made**: `yaml.safe_load()` to produce a Python data structure, then recursive `_extract_yaml_file_refs()` traversal (Option 2)

**Rationale**: Structural traversal visits every string value regardless of YAML formatting style. No false positives from YAML syntax. Works for arbitrarily nested structures.

**Implications**:

- YAML comments are stripped by safe_load and cannot be parsed â€” accepted trade-off
- Large YAML files are loaded fully into memory before traversal

---

### Decision 2: GenericParser Fallback for Invalid YAML

**Date**: 2026-02-18 (discovered)
**Context**: YAML files may have syntax errors or non-standard extensions. A parse failure must not silently discard all link references.

**Options Considered**:

1. Return empty list on error: Simple but causes silent data loss
2. GenericParser fallback (chosen): Best-effort quoted-string extraction from raw text
3. Re-raise exception: Breaks monitoring for a single bad file

**Decision Made**: Catch `yaml.YAMLError`; delegate to `GenericParser` (Option 2)

**Rationale**: Consistent with JSON parser pattern. Recovers partial reference data from malformed files. Monitoring continues uninterrupted.

---

### Decision 3: _find_next_occurrence() for Duplicate Value Position Tracking

**Date**: 2026-02-18 (discovered)
**Context**: After `yaml.safe_load()`, each extracted string value must be mapped back to its line/column in the raw file. When the same string appears multiple times, a simple text search would always return the first occurrence.

**Options Considered**:

1. Always search from start of file: Returns wrong position for 2nd+ occurrences
2. Track scan offset; advance past each match (chosen): Each occurrence found at its correct position
3. YAML Loader line-number metadata: Complex; not available with safe_load

**Decision Made**: `_find_next_occurrence(content, value, start_offset)` advancing offset after each match (Option 2)

**Rationale**: Guarantees correct line/column for all occurrences of duplicate path strings. Implemented as a reusable `BaseParser` utility.

### Implementation Patterns Used

**Structural Pattern**:

- Pattern: Template Method (via `BaseParser`)
- Why: Common file-read infrastructure in base; YAML-specific traversal in `YamlParser`
- Where: `BaseParser._safe_read_file()`; `YamlParser._extract_yaml_file_refs()`

**Resilience Pattern**:

- Pattern: Fallback / Graceful Degradation
- Why: A single malformed YAML file must not halt monitoring or lose all extraction
- Where: `yaml.YAMLError` catch â†’ `GenericParser.extract_links()`
