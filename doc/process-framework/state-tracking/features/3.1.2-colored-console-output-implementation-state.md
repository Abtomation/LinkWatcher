## 1. Feature Overview

### Feature Description

Provides visually distinct console log output using ANSI colors and emoji icons, mapped per log level. Implemented as `ColoredFormatter` class in `logging.py`, extending `logging.Formatter`. Uses the `colorama` library with `init(autoreset=True)` for cross-platform Windows/Linux/macOS terminal support.

Maps five log levels to distinct colors (DEBUG=Cyan, INFO=Green, WARNING=Yellow, ERROR=Red, CRITICAL=Magenta+Bright) and emoji icons. The formatter also injects thread-local context variables from `LogContext` into the output, filtering out stdlib record attributes to show only user-defined context.

### Business Value

- **User Need**: Quickly distinguish log severity at a glance in terminal output during development and debugging
- **Business Goal**: Reduce time-to-diagnosis by making errors and warnings visually prominent
- **Success Metrics**: Clear visual distinction between log levels; context displayed inline with messages

### Scope

**In Scope**:

- Per-level color mapping using colorama ANSI codes
- Per-level emoji icon mapping
- Configurable `colored` and `show_icons` boolean flags
- Thread-local context injection into formatted output
- Automatic color reset via `autoreset=True`

**Out of Scope**:

- Custom color themes or user-configurable color mappings
- Rich/TUI-style output (uses plain ANSI codes only)

## 7. Design Decisions

### Decision 1: colorama Over Raw ANSI Codes

**Context**: Need colored terminal output that works on Windows, Linux, and macOS.

**Decision Made**: Use `colorama` library with `init(autoreset=True)`.

**Rationale**: Raw ANSI codes don't work on Windows cmd.exe. colorama translates ANSI sequences to Windows API calls. `autoreset=True` prevents color leaking between log lines.

**Implications**:
- External dependency (colorama) but very lightweight and widely used
- `init()` called at module import time â€” affects all console output in the process

### Decision 2: Dict-Based Color/Icon Mapping

**Context**: Need to map 5 log levels to colors and icons.

**Decision Made**: Static `COLORS` and `ICONS` class-level dicts with string keys matching `record.levelname`.

**Rationale**: Simple, maintainable, O(1) lookup. Adding new levels requires only a dict entry. Fallback to empty string for unknown levels.

### Decision 3: Context-Aware Formatting with stdlib Attribute Filtering

**Context**: Thread-local context should appear in log output, but not duplicate stdlib record fields.

**Decision Made**: Merge `LogContext` into formatted output, filtering a hardcoded list of stdlib `LogRecord` attribute names.

**Rationale**: Prevents internal logging metadata from appearing in user-visible context. The filter list covers all known stdlib fields.

**Implications**:
- Hardcoded filter list may need updates if Python stdlib adds new LogRecord attributes
- Context displayed as `[key=value, ...]` inline after the message

---

### Implementation Patterns Used

**Template Method Pattern**:
- Pattern: Override `format(record)` on `logging.Formatter`
- Why: Plug into stdlib handler pipeline with custom formatting
- Where: `ColoredFormatter.format()`

**Lookup Table Pattern**:
- Pattern: Dict constants for color/icon mapping
- Why: Simple, efficient, easily extensible mapping
- Where: `COLORS` and `ICONS` class attributes
