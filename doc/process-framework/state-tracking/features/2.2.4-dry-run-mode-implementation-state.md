## 1. Feature Overview

### Feature Description

Dry-run mode is controlled by a `dry_run` flag on `LinkUpdater` (default `False`). When enabled, `_update_file_references()` logs `[DRY RUN] Would update X reference(s) in {file_path}` and returns `True` immediately â€” no file reads or writes occur. Statistics counters (`files_updated`, `references_updated`) are still incremented as if the update had run, giving an accurate count of what would change. The flag is set at construction time via the `dry_run` parameter or changed at runtime via `set_dry_run(enabled: bool)`.

`LinkWatcherService` exposes dry-run mode through its own `set_dry_run()` method, which passes the flag through to the `LinkUpdater` instance. The CLI `--dry-run` argument activates the flag when starting the watcher via `main.py`, enabling safe preview of all pending changes without any file modification.

### Business Value

- **User Need**: Users need to verify what would change before committing to file modifications across their project
- **Business Goal**: Provide a safe preview mode showing the full scope of pending reference updates without touching files
- **Success Metrics**: Zero file I/O in dry-run mode; statistics accurately report the number of pending changes; `[DRY RUN]` clearly labels all log output

### Scope

**In Scope**:

- `dry_run` constructor parameter and `set_dry_run()` API on `LinkUpdater`
- Early return from `_update_file_references()` with `[DRY RUN]` log entry
- Statistics accumulation (`files_updated`, `references_updated`) even in dry-run mode
- CLI `--dry-run` flag propagation through service layer

**Out of Scope**:

- Diff output showing exact line changes
- Report file generation
- Interactive confirmation prompts

## 7. Design Decisions

### Decision 1: Flag-Based Early Return with Statistics Still Accumulated

**Date**: 2026-02-18 (discovered)
**Context**: Dry-run mode must accurately preview the scope of pending changes without performing any file I/O. Simply blocking writes is insufficient if statistics are not accumulated â€” users need to know how many files and references would be affected.

**Options Considered**:

1. Skip all processing in dry-run: No statistics accumulated; user gets no information about scope
2. Full processing with write blocked at the last step: Wasteful; reads files unnecessarily
3. Early return after logging, with statistics still incremented (chosen): Zero file I/O; full statistics preview

**Decision Made**: Log `[DRY RUN] Would update...`, increment `files_updated` and `references_updated`, then return `True` immediately without reading or writing (Option 3)

**Rationale**: Statistics are the primary output of dry-run mode. Skipping them defeats the purpose. Early return is efficient â€” no file reads or writes occur.

**Implications**:

- Statistics in dry-run reflect what would happen, not what did happen â€” clearly labelled in log output
- Error counter is not incremented in dry-run (errors can only be discovered during actual file operations)

---

### Decision 2: set_dry_run() Runtime API

**Date**: 2026-02-18 (discovered)
**Context**: Dry-run mode may need to be toggled after `LinkUpdater` is instantiated â€” for example, when the CLI receives a signal or the service layer reads configuration at startup.

**Options Considered**:

1. Constructor-only `dry_run` parameter: Cannot change mode after instantiation; incompatible with service-layer pattern
2. `set_dry_run(enabled: bool)` runtime setter (chosen): Allows toggling at any point; consistent with `set_backup_enabled()`
3. Both constructor and runtime setter (implemented): Constructor for initial state; setter for runtime changes

**Decision Made**: Both constructor parameter `dry_run=False` and `set_dry_run(enabled: bool)` setter (Option 3)

**Rationale**: Constructor default ensures safe initial state. Runtime setter enables configuration-driven activation after instantiation. Symmetric with `set_backup_enabled()` for API consistency.

---

### Implementation Patterns Used

**Behavioural Pattern**:

- Pattern: Feature Flag (boolean toggle on `LinkUpdater` instance)
- Why: Dry-run is a mode of operation, not a separate class; flag cleanly toggles behaviour at a single decision point
- Where: `LinkUpdater.dry_run` flag; checked in `_update_file_references()`
