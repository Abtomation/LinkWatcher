## 1. Feature Overview

### Feature Description

Provides structured error reporting with severity levels, exception tracebacks, and visual highlighting. The `LinkWatcherLogger` exposes `error()`, `critical()`, and `exception()` methods that delegate to structlog for structured error output. The `exception()` method automatically includes traceback information.

The `LogTimer` context manager enhances error reporting by logging `failed_*` messages with `error_type` and `error_message` kwargs when an exception occurs within a timed block. Backward compatibility functions (`log_error()`, `log_warning()`) delegate to the global singleton, and the `ColoredFormatter` applies red (ERROR) or magenta+bright (CRITICAL) ANSI colors for visual prominence.

### Business Value

- **User Need**: Quickly identify and diagnose errors when file operations fail
- **Business Goal**: Minimize debugging time by providing structured error context with severity, component, and traceback
- **Success Metrics**: All errors include structured context; exceptions include tracebacks; visual distinction in console

### Scope

**In Scope**:

- `error()`, `critical()`, `exception()` methods on `LinkWatcherLogger`
- `LogTimer.__exit__` error logging with error_type and error_message
- Backward compat functions: `log_error()`, `log_warning()`
- Color-coded error output (RED for ERROR, MAGENTA+BRIGHT for CRITICAL)
- JSON-formatted error entries in file logs with `exception` field

**Out of Scope**:

- Error aggregation or deduplication
- External error reporting services (Sentry, etc.)
- Error recovery or retry logic (handled by calling code)

## 7. Design Decisions

### Decision 1: structlog exception() for Automatic Traceback

**Context**: Errors need traceback information for debugging but manual `traceback.format_exc()` is verbose and error-prone.

**Decision Made**: Delegate to `structlog.exception()` which automatically captures and formats traceback.

**Rationale**: structlog handles `exc_info` extraction and formatting. No need for manual traceback management. Works consistently across all error paths.

### Decision 2: LogTimer Error Reporting with Structured Context

**Context**: When timed operations fail, need to know both the error and the operation context.

**Decision Made**: `LogTimer.__exit__` logs `failed_{operation}` with `error_type=exc_type.__name__` and `error_message=str(exc_val)` as structured kwargs.

**Rationale**: Structured fields enable filtering/searching errors by type. Operation name is embedded in the event key (`failed_scan`, `failed_update`). Does not suppress the exception (returns `None`/falsy from `__exit__`).

### Decision 3: Backward Compatibility Functions

**Context**: Legacy code uses `log_error()`, `log_warning()` etc. as module-level functions.

**Decision Made**: Provide `log_error()`, `log_warning()`, `log_info()`, `log_debug()` that delegate to `get_logger()`.

**Rationale**: Zero-cost migration path. New code uses `get_logger().error()` directly; legacy code continues working unchanged.

---

### Implementation Patterns Used

**Delegation Pattern**:
- Pattern: Backward compat functions delegate to singleton logger
- Why: No code duplication; single implementation point
- Where: `log_error()` â†’ `get_logger().error()`, etc.

**Structured Error Context Pattern**:
- Pattern: Error kwargs include `error_type` and `error_message` as separate fields
- Why: Enables filtering/grouping errors by type in log analysis tools
- Where: `LogTimer.__exit__()` error logging
