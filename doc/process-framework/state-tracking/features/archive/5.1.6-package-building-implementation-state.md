## 1. Feature Overview

### Feature Description

Configures Python package building and distribution for LinkWatcher. The project uses a dual-configuration approach: `pyproject.toml` (PEP 621 standard â€” primary) and `setup.py` (legacy compatibility). Both define the same metadata: name=linkwatcher, version=2.0.0, Python >=3.8, runtime dependencies (watchdog, PyYAML, colorama), optional dependency groups (`[test]` and `[dev]`), entry point (`linkwatcher=linkwatcher.cli:main`), and classifiers.

The CI `build` job runs `python -m build` + `twine check dist/*` on push events, gated behind the `test` and `quality` jobs. Deployment scripts (`deployment/install_global.py`, `deployment/setup_project.py`) handle local installation. The `Makefile` provides `clean`, `build`, and `release-check` targets. Supporting files include `requirements.txt`, `.gitignore`, and `CHANGELOG.md`.

### Business Value

- **User Need**: Install LinkWatcher as a standard Python package via pip
- **Business Goal**: Distributable package with proper metadata, dependencies, and entry points
- **Success Metrics**: `python -m build` succeeds; `twine check` passes; `pip install -e .` works

### Scope

**In Scope**:

- `pyproject.toml` â€” build system, project metadata, dependency groups, tool configs
- `setup.py` â€” legacy setuptools configuration (mirrors pyproject.toml)
- `deployment/install_global.py`, `deployment/setup_project.py` â€” installation scripts
- `requirements.txt`, `.gitignore`, `CHANGELOG.md`, `MULTI_PROJECT_SETUP.md` â€” supporting files
- CI `build` job in `ci.yml`

**Out of Scope**:

- CI pipeline structure (covered by 5.1.1)
- Tool configuration sections in pyproject.toml (covered by 5.1.3)
- Makefile/dev.bat build targets (covered by 5.1.7)

## 7. Design Decisions

### Decision 1: Dual Configuration (pyproject.toml + setup.py)

**Context**: Python packaging supports PEP 621 (pyproject.toml) as the modern standard, but some tools still expect setup.py

**Decision Made**: Maintain both `pyproject.toml` (primary) and `setup.py` (legacy compatibility) with identical metadata

**Rationale**: `pyproject.toml` is the modern standard and supports all PEP 621 metadata. `setup.py` is kept for backwards compatibility with older pip versions and tools that don't support PEP 621. Both define the same version, dependencies, and entry points.

**Implications**:

- Metadata must be kept in sync between both files (duplication risk)
- Supports widest range of installation methods
- Can drop setup.py when minimum pip version is >=21.3

### Decision 2: Optional Dependency Groups

**Context**: Different users need different dependency sets (runtime, testing, development)

**Decision Made**: Define `[test]` and `[dev]` optional dependency groups in both pyproject.toml and setup.py

**Rationale**: Separation allows production installs to skip dev/test dependencies. `pip install -e ".[dev]"` installs everything; `pip install .` installs only runtime deps.

**Implications**:

- Clean production installs without test/dev overhead
- CI can install specific groups as needed
- Dev setup includes all groups via `dev-setup` target
