## 1. Feature Overview

### Feature Description

Defines pre-commit hook configuration in `.pre-commit-config.yaml` that runs code quality checks automatically before each commit. The configuration includes 4 hook repositories: `pre-commit-hooks` (v4.4.0 â€” trailing whitespace, end-of-file fixer, YAML/JSON validation, merge conflict detection, large file check, debug statement detection), `black` (v23.7.0 â€” code formatting with `--line-length=100`), `isort` (v5.12.0 â€” import sorting with `--profile=black`), and `flake8` (v6.0.0 â€” linting with `--max-line-length=100`).

Additionally, a local `pytest-quick` hook runs `python run_tests.py --quick` at the `commit` stage with `always_run: true`, ensuring quick tests pass before any commit is accepted. Installation is done via `pre-commit install` and `pre-commit install --hook-type commit-msg`, automated through `scripts/setup_cicd.py` and Makefile/dev.bat `pre-commit` targets.

### Business Value

- **User Need**: Catch formatting, linting, and test failures before they enter version control
- **Business Goal**: Shift quality checks left; reduce CI failures from preventable issues
- **Success Metrics**: All hooks pass before commits; no formatting or import order issues in PRs

### Scope

**In Scope**:

- `.pre-commit-config.yaml` â€” hook definitions (4 external repos + 1 local)
- Pre-commit installation via setup scripts (Makefile, dev.bat, setup_cicd.py)

**Out of Scope**:

- CI-level quality checks (covered by 5.1.3)
- Tool configuration details in pyproject.toml (covered by 5.1.3)

## 7. Design Decisions

### Decision 1: Local Test Hook at Commit Stage

**Context**: Pre-commit hooks typically run formatting/linting only; should tests also run before commit?

**Decision Made**: Include a local `pytest-quick` hook that runs `python run_tests.py --quick` with `always_run: true` at the `commit` stage

**Rationale**: Quick tests (unit + parsers) are fast enough to run on every commit. This catches obvious regressions before they enter version control, reducing CI feedback time. Full test suite runs in CI.

**Implications**:

- Commits take slightly longer (quick test execution time)
- Developers get immediate local feedback on breaking changes
- `always_run: true` means tests run even if no Python files changed

### Decision 2: Pinned Hook Repository Versions

**Context**: Pre-commit hooks can be pinned to specific versions or use latest

**Decision Made**: Pin all external hooks to specific versions (e.g., `rev: v4.4.0`, `rev: 23.7.0`)

**Rationale**: Pinned versions ensure reproducible behavior across all developer machines and CI. `pre-commit autoupdate` can be used periodically to update versions.

**Implications**:

- Consistent behavior regardless of when hooks are installed
- Requires periodic manual updates to get new features/fixes
- Version alignment with CI tool versions (same black/isort/flake8 versions)
