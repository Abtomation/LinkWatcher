## 1. Feature Overview

### Feature Description

Provides static test data files and manual test project structures used across the test suite. The `tests/fixtures/` directory contains sample data files: `sample_markdown.md` (markdown with various link types), `sample_config.yaml` (YAML with file references), and `sample_data.json` (JSON with file paths). These fixtures provide stable, version-controlled test inputs.

Additionally, the `manual_markdown_tests/` directory contains 24 markdown test case files (LR-001 through MP-009) with a `test_runner.py` for interactive parser validation. The `manual_test/` directory provides a complete test project structure (13 files) with docs, src, assets, and scripts subdirectories for end-to-end manual testing of LinkWatcher against a realistic project layout.

### Business Value

- **User Need**: Stable, reproducible test data that covers diverse link formats and project structures
- **Business Goal**: Eliminate test flakiness from dynamically generated data; provide manual testing targets
- **Success Metrics**: Fixture files cover all supported file types; manual test project mirrors real-world structure

### Scope

**In Scope**:

- `tests/fixtures/` â€” Static sample files (markdown, YAML, JSON)
- `manual_markdown_tests/` â€” 24 markdown test cases with interactive runner
- `manual_test/` â€” Complete test project structure (13 files)
- `tests/test_config.py` sample content constants (`SAMPLE_CONTENTS`, `TEST_PROJECT_STRUCTURES`)

**Out of Scope**:

- Dynamically generated test data (covered by 4.1.7 via builders)
- pytest fixtures (covered by 4.1.1)

## 7. Design Decisions

### Decision 1: Static Fixture Files for Parser Verification

**Context**: Parser tests need consistent, known-good input files to validate parsing behavior

**Decision Made**: Provide pre-written fixture files (`sample_markdown.md`, `sample_config.yaml`, `sample_data.json`) in `tests/fixtures/` rather than generating all test data dynamically

**Rationale**: Static fixtures guarantee deterministic, human-readable test inputs. Easier to inspect and debug when parser tests fail. Dynamic generation (via `TestProjectBuilder`) is used alongside for complex scenarios.

**Implications**:

- Parser tests can reference fixed file content for exact-match assertions
- Fixture files must be maintained if supported formats change
- Complements (not replaces) programmatic test data from `tests/utils.py`

### Decision 2: Manual Test Project Scripts

**Context**: Some testing scenarios (e.g., real file-move watching) require a physical project structure on disk that persists across manual test runs

**Decision Made**: Provide `scripts/create_test_structure.py` and `scripts/cleanup_test.py` for manual test project lifecycle management

**Rationale**: Automated tests use temporary directories; manual/exploratory testing needs persistent structures. Separate scripts keep this concern out of the pytest test suite.

**Implications**:

- Manual testing has a reproducible setup path
- Cleanup script prevents stale test directories from accumulating
