## Improvement Suggestions

### What worked well
- New-E2EAcceptanceTestCase.ps1 automation eliminated manual state tracking updates entirely
- Cross-cutting spec's coverage gap table made it immediately clear what needed to be created
- Existing TE-E2E-009 test case provided a strong pattern to follow for fixture design

### What could be improved
- Template `feature_ids` frontmatter field is generated as a single-string array element when multiple feature IDs are passed comma-separated — should be parsed into separate array elements
- **New-E2EAcceptanceTestCase.ps1 does not update Workflow Milestone Tracking table** in test-tracking.md — WF-002 E2E Cases column still shows `—` instead of listing TE-E2E-009, TE-E2E-011, TE-E2E-013, TE-E2E-014. Workflows WF-003 through WF-008 also show `⬜ Not Created` despite having cases created.
- **New-E2EAcceptanceTestCase.ps1 does not create E2E Group entries** (TE-E2G-004 through TE-E2G-011) in the main E2E Test Cases table. Only TE-E2G-001 through TE-E2G-003 exist (created manually in earlier sessions). Groups for yaml-json-python, runtime-dynamic-operations, startup-operations, rapid-sequential-moves, multi-format-references, dry-run-mode, graceful-shutdown, and error-recovery are all missing.

### Specific suggestions
- Fix `feature_ids` frontmatter generation in New-E2EAcceptanceTestCase.ps1 to produce `["1.1.1", "0.1.2", "2.1.1", "2.2.1"]` instead of `["1.1.1, 0.1.2, 2.1.1, 2.2.1"]`
- New-E2EAcceptanceTestCase.ps1 should update the Workflow Milestone Tracking table: add the test case ID to the E2E Cases column and update the Status from `⬜ Not Created` to `📋 Cases Created` when first case is added for a workflow
- New-E2EAcceptanceTestCase.ps1 should auto-create the TE-E2G group entry in the E2E Test Cases table when creating the first test case for a new group (or when the group entry doesn't exist yet)
