## 1. Feature Overview

### Feature Description

Link Validation is an on-demand workspace scanner that checks all existing links across all supported file formats (Markdown, YAML, JSON, Python, Dart, PowerShell, generic) and reports broken references — targets that no longer exist on disk. It reuses the existing parser infrastructure (2.1.1) to extract links and the path resolution utilities (0.1.1) to resolve and verify link targets.

Unlike the existing real-time link maintenance system (which reacts to file moves), Link Validation proactively audits the current state of all links in the workspace, identifying references that have become stale through manual edits, external tooling, or other changes not caught by the file watcher.

### Business Value

- **User Need**: Confidence that all cross-references in the workspace are valid, especially after bulk operations or when starting to use LinkWatcher on an existing project
- **Business Goal**: Complete link health visibility with a single command
- **Success Metrics**: All broken links in workspace detected and reported with file location and line number

### Scope

**In Scope**:

- Scanning all files matching LinkWatcher's configured file extensions
- Extracting links using existing parsers
- Resolving relative paths and checking if targets exist on disk
- Reporting broken links with source file, line number, and target path
- CLI invocation (e.g., `python main.py --validate`)

**Out of Scope**:

- URL validation (HTTP/HTTPS links) — local file references only
- Automatic fixing of broken links
- Integration with the real-time watcher (this is a separate on-demand tool)

## 8. Design Decisions

### Decision 1: Output file location derived from log-file path

**Date**: 2026-03-24
**Context**: Need to determine where to write `LinkWatcherBrokenLinks.txt`

**Options Considered**:
1. Dedicated `--output` CLI flag — flexible but adds a new parameter
2. Derive from `--log-file` / `config.log_file` parent directory — consistent with existing run directory pattern
3. Always write to project root — simple but clutters project root

**Decision Made**: Option 2 — derive from log file location

**Rationale**: The run directory is already implicitly defined by the log file path. The startup script passes `--log-file LinkWatcher_run/LinkWatcherLog.txt`, so the report naturally lands in the same directory. Fallback to project root when no log file is configured.

**Resolution order**: `--log-file` CLI arg → `config.log_file` → project root

### Decision 2: Skip Python imports in v1

**Date**: 2026-03-24
**Context**: Python import targets use dot-separated module paths (e.g., `app.utils`), not file paths

**Decision Made**: Skip `link_type == "python-import"` targets entirely

**Rationale**: Converting module paths to file paths requires understanding Python's import resolution (packages, `__init__.py`, relative imports). Out of proportion for v1. Can be added as enhancement later.

### Implementation Patterns Used

**Data Flow Pattern**:
- Pattern: Walk → Parse → Filter → Resolve → Check → Collect → Report
- Why: Simple pipeline, no state management needed
- Where: `LinkValidator.validate()`
