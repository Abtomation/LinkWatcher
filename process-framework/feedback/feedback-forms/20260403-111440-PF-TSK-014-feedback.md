The Structure Change task needs a much more thorough research/impact analysis phase **before** the proposal is presented for review. In this session, the human partner had to repeatedly catch missing scope:
1. First checkpoint missed scripts that write to the documentation map
2. Second checkpoint missed that task definitions instruct agents to manually update the map
3. Naming convention (PF-/PD-/TE- prefixes) was only caught during execution
4. The need to consider LinkWatcher-missed links was raised during execution

The task definition should require the AI agent to:
- **Systematically grep for all references** to affected files (not just read the proposal template sections)
- **Check all scripts** that read from or write to affected files
- **Check all task definitions** for manual update instructions referencing affected files
- **Consult framework infrastructure docs** like `process-framework-task-registry.md` (which catalogs exactly what each task creates/updates) and `task-transition-guide.md` (which documents handover interfaces between tasks) — these would have surfaced most of the missed scope items immediately
- **Present a complete impact matrix** (files × change type) before the proposal review, not discover gaps incrementally through human corrections

The initial scope assessment (Lightweight) was also wrong — a file that 126 other files reference should never be classified as Lightweight regardless of other criteria.
