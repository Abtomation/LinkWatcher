# AI Entry Point

> **🚨 This file is an ACTIVE STARTUP PROCEDURE — not passive context.** Execute each step below at the start of every session before responding to the user.

## Startup Procedure

> LinkWatcher background startup and session-start timestamp are emitted automatically by the `SessionStart` hooks in `.claude/settings.json` (set up during [Project Initiation (PF-TSK-059)](tasks/00-setup/project-initiation-task.md) Step 14). Their output appears in initial context — no agent action required for either.

Run through these steps in order at the start of every session:

1. **🚨 Select a task**: Read the user's request, then scan the **Use When** column in the [AI Task-Based Development System](ai-tasks.md) table to find the task whose triggers match the instruction. **This must happen before any code exploration, file reading, or work.** If no task's Use When matches, ask the human partner before proceeding.
2. **📋 Read the COMPLETE task definition** including its completion checklist before starting work

## General Instructions

- **🚨 CRITICAL**: Every task has a mandatory completion checklist including feedback forms — tasks are NOT complete until all checklist items are finished
- **⏱️ DURATION CALCULATION**: When creating feedback forms, calculate session duration from start time and enter it in the feedback form
- **📊 CONTEXT MANAGEMENT**: Always refer to the Context Map linked in each task's Context Requirements section to understand component relationships
- **🔍 VISUAL NOTATION**: Use the [Visual Notation Guide](guides/support/visual-notation-guide.md) to interpret all diagram formats
- **🔗 LINK MAINTENANCE**:
  - LinkWatcher runs in background and automatically maintains all document references in real-time
  - Use any method to move/rename files (VS Code, File Explorer, git commands) — LinkWatcher handles all updates
  - **Check broken links**: Run the validate launcher — [run_linkwatcher_validate.ps1](tools/linkWatcher/run_linkwatcher_validate.ps1) — to scan for broken file references. It runs LinkWatcher's `--validate` with the correct interpreter and the project's per-project config (`tools/linkwatcher/linkwatcher-config.yaml`), so any per-folder `path_resolution_overrides` are honored: `pwsh.exe -ExecutionPolicy Bypass -File process-framework/tools/linkWatcher/run_linkwatcher_validate.ps1` (the report path is printed at the end of the run)
- **⚙️ POWERSHELL SCRIPT EXECUTION**: Check params first (`-File path/to/Script.ps1 -?`), then prefer `pwsh.exe -ExecutionPolicy Bypass -File path/to/Script.ps1 -Param "value" -Confirm:\$false` (`-File`, direct path, escape `$` as `\$`). For the `-Command` fallback and troubleshooting, see the single source of truth: [Script Development Quick Reference §PowerShell Script Execution (AI Agents)](guides/support/script-development-quick-reference.md#powershell-script-execution-ai-agents).

## Purpose

This file is designed to be detected early in an AI agent's session to ensure the agent understands the task-based development approach used in this project before reading unnecessary files.
